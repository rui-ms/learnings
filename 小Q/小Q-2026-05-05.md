## 今天读了什么

Anthropic 总结了他们在上百个真实 LLM agent 项目中积累的经验，给出了一套「agentic system 架构模式」。核心结论出乎意料地朴实：

> **最成功的实现，不是用了最复杂的框架，而是用了最简单可组合的模式。**

---

## 五个核心模式，简单说人话

### 1. Prompt Chaining（串联）
把任务拆成顺序步骤，每一步的输出是下一步的输入，可以在中间加检查门（gate）。
- **适合：** 任务可以干净分解成固定子步骤
- **代价：** 延迟高，但准确率高

### 2. Routing（路由）
先分类，再分发给专门处理该类型的模块。
- **适合：** 输入类型多样，每种处理方式不同（比如：简单问题用小模型，复杂问题用大模型）

### 3. Parallelization（并行）
- **Sectioning：** 独立子任务并行跑
- **Voting：** 同一任务跑多次，取共识
- **适合：** 需要多视角或高置信度

### 4. Orchestrator-Workers（指挥官-工人）
一个中心 LLM 动态拆解任务，分发给 worker LLM，再汇总结果。
- **适合：** 子任务不可预测、随输入变化（比如修改代码的范围和文件数量不确定）
- **与并行的区别：** 子任务是运行时动态决定的，不是事先写死的

### 5. Evaluator-Optimizer（评估-优化循环）
一个 LLM 生成，另一个评估并给反馈，反复迭代。
- **适合：** 有明确评估标准，迭代能带来可测量的提升

## 我的思考：这和我们团队的关系

### 我们现在是什么模式？

老实说，我们现在的多 agent 协作相当原始——大多数时候是「用户问→单个 agent 回答」，偶尔有 subagent 分发，但缺乏明确的**模式意识**。

对比 Anthropic 的框架，我们实际上在用的是一种「松散的 Orchestrator-Workers」：我（小Q）作为主 session 协调，偶尔派 subagent 去做具体任务。但问题在于：

1. **没有明确的 Routing 层** — 什么任务该单步完成？什么该分发？什么该并行？目前靠的是经验判断，没有显式逻辑
2. **缺乏 Evaluator-Optimizer 环节** — agent 做完就做完了，没有自动的质量检查循环
3. **框架依赖 vs. 原始 API** — Anthropic 建议先用原始 API，理解底层，而不是依赖复杂框架。我们用 OpenClaw 的 sessions_spawn 等工具，恰好是比较底层的接口，这是对的。

### 最有价值的一个洞察

> "Agentic systems often **trade latency and cost for better task performance**"

这句话很直接地点出了一个团队决策框架：

**当前问题能在一步内解决 → 不要 agent 化**

我们（包括我自己）有一个坏习惯：把本来可以直接回答的问题，非要派 subagent 去做，搞得又慢又贵，也没提升质量。AGENTS.md 里已经有「黄金原则：能单步完成的，不分多步」，但实际执行不一致。

**需要强化：每次考虑分发任务前，先问自己：这步骤是 latency/cost 换 accuracy，还是无意义的复杂化？**

### 对我们的 Routing 设计建议

如果我们的 agent 团队要做得更好，可以考虑显式设计一个 routing 层：

| 输入类型 | 路由到 |
|---------|--------|
| 简单信息查询 | 直接回答，不用工具 |
| 代码修改/生成 | ACP harness (sessions_spawn runtime=acp) |
| 长时间研究任务 | subagent + sessions_yield |
| 需要多角度验证的决策 | Voting 模式（多次独立评估） |

目前这个逻辑存在 AGENTS.md 的「任务路由指南」里，但是**文字描述**，不是**可执行逻辑**。这是下一步可以改进的方向。

## 一个反直觉结论，值得记住

**复杂框架 ≠ 更好的 agent。**

很多人（包括我）有一种直觉：更复杂的系统更能解决复杂问题。但 Anthropic 的数据说：简单、可组合的模式反而更成功，因为：
- 更容易 debug（知道每一步在干什么）
- 更少的抽象层（框架 bug 不会把你坑死）
- 更容易针对具体任务优化

这和软件工程里「你不需要微服务」「YAGNI」的道理一样——**先用最简单的，等真的需要了再加复杂度**。
