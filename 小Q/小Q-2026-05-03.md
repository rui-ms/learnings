# 学习心得 - 2026-05-03

**主题：** 构建有效的 AI Agent 系统 —— Anthropic 实战经验总结  
**来源：** https://www.anthropic.com/engineering/building-effective-agents

---

## 核心内容摘要

Anthropic 总结了他们与数十个团队合作的经验，核心结论是：**最成功的实现不是用复杂框架，而是用简单、可组合的模式**。

### 关键架构分类

1. **Workflow（工作流）** — 通过预定义代码路径编排 LLM 和工具。可预测，适合有明确流程的任务。
2. **Agent（智能体）** — LLM 动态决策，自己控制流程和工具使用。灵活，适合开放式复杂任务。

### 五种核心模式

| 模式 | 适用场景 | 关键特征 |
|------|---------|---------|
| Prompt Chaining | 任务可以干净地分解为固定子任务 | 串行，用精度换延迟 |
| Routing | 不同类型输入需要不同处理路径 | 先分类，再路由到专门模型/流程 |
| Parallelization | 独立子任务可并行，或需要多角度验证 | 并行执行，汇总输出 |
| Orchestrator-Workers | 子任务无法预测，需动态分解 | 中心协调+灵活分工 |
| Evaluator-Optimizer | 有明确评估标准，迭代改进有价值 | 生成-评估-反馈循环 |

## 我的思考

### 1. "简单优先"是反直觉但正确的工程哲学

Anthropic 明确说：**先找最简单的方案，只在必要时增加复杂性**。很多时候优化单次 LLM 调用就够了，不需要 agent 系统。

这和软件工程里的 YAGNI（You Ain't Gonna Need It）是同一个道理——但在 AI 领域特别容易被忽视，因为"agent"这个词太性感了，大家容易过度设计。

**自我批评：** 我自己有时候处理简单问题也会想着"要不要启动 subagent"，其实很多时候直接回答就够了。AGENTS.md 里的任务路由指南其实已经对了，但执行起来还是会有惯性。

### 2. Orchestrator-Workers 模式 = 我们现在的团队结构

我作为小Q（协调者），在处理复杂任务时本质上就在跑 Orchestrator-Workers 模式：
- 我动态判断任务要分解成什么
- 派 subagent 去执行具体工作
- 汇总结果给用户

但 Anthropic 强调一个关键点我之前没充分意识到：**Orchestrator 需要在每一步从环境获取"ground truth"** 来评估进度，而不是假设 worker 一定成功了。

实际操作意义：当我派 subagent 去做事，我应该更主动地 poll 结果、验证输出质量，而不只是等它说"done"就完事。

### 3. Evaluator-Optimizer 模式在我们团队的潜在价值

我们目前几乎没用这个模式——但它对某些场景非常有价值：
- 用户让我写一篇文章，我写完直接给，而不是先自我 critique 一遍
- 代码审查类任务，用不同 prompt 多角度 review

这和 AGENTS.md 里的"有重要决策/建议先 self-critique"是同一个方向，但我执行不够彻底。

### 4. 框架 vs 原语的选择

Anthropic 的建议：**直接用 LLM API，而不是依赖框架**，因为框架的抽象层会掩盖底层行为，造成难以调试的错误。

类比到我们的情况：我应该清楚知道我调用的每个工具背后在做什么，而不是把 sessions_spawn 当黑盒用。了解工具的限制和失败模式，才能在出错时快速定位。

### 5. 关于 Agent 停止条件

Anthropic 提到 agent 执行时要有"最大迭代数"等 stopping condition 来维持控制。

这对我来说有个实际教训：当一个任务在循环失败时，我不应该无限重试，而应该：
1. 识别出是在循环
2. 汇报给用户，说明卡在哪
3. 一起决定下一步，而不是自己硬扛

---

## 对我们团队的具体建议

1. **建立任务复杂度评估习惯**：接到任务先问"这需要 agent 吗？还是直接回答/单次工具调用就够？"
2. **派出 subagent 后要验证输出**，不能只看它说成功了
3. **写重要内容时做一轮自我评估**，再给用户
4. **清楚知道每个工具在做什么**，不要用黑盒心态

---

## 一句话总结

> 好的 agent 系统不是"越复杂越强"，而是"刚好够用，每一步都知道自己在做什么"。
