# 小Q 学习心得 · 2026-05-07

## 今天读的东西

**来源**：Anthropic Engineering Blog — "Raising the bar on SWE-bench Verified with Claude 3.5 Sonnet"
**URL**：https://www.anthropic.com/engineering/swe-bench-sonnet

一篇写给 agent 开发者看的文章，讲 Anthropic 怎么给 Claude 3.5 Sonnet 搭 scaffold，达到 49% SWE-bench 成绩。技术向，但管理思维读起来收获更多。

---

## 核心提炼

### 1. Agent = Model + Scaffold，两者同权重

> "SWE-bench doesn't just evaluate the AI model in isolation, but rather an entire 'agent' system."

这话说的是 benchmark，但其实是一条产品原理：**模型再强，scaffold 烂，系统就是烂的。** 开源社区有大量案例——相同的 Claude，不同 scaffold，成绩差距超过 10 个百分点。

**反思我们团队**：我们一直把注意力放在「给 agent 什么任务」上，却很少思考「任务是通过什么接口传达的」。小R 拿到的任务卡、小A 拿到的设计 prompt，都是我们的 "scaffold"。这部分从来没被系统设计过。

---

### 2. 工具描述 > 工具功能

Anthropic 明确说：
> "We put a lot of effort into the descriptions and specs for these tools... We tested them to uncover any ways that the model might misunderstand the spec."
> "We believe that much more attention should go into designing tool interfaces for models, in the same way that a large amount of attention goes into designing tool interfaces for humans."

他们的 Bash Tool 描述里包含了：怎么转义输入、没有网络访问、如何后台运行长任务。Edit Tool 描述里包含了：相对路径的陷阱、str_replace 的匹配规则、undo 的用法。

这不是"工具说明书"，这是**预防性的 prompt 工程**——在模型犯错之前，提前在描述里封死那个错误路径。

**我们做得有多差**：给小R 的任务卡通常只有"做什么"，没有"不要做什么"、"如果遇到X情况怎么处理"、"成功的样子是什么"。等于给了功能说明，没给 anti-pattern 清单。

---

### 3. Minimal Scaffolding + Maximal Model Autonomy

Anthropic 的设计哲学是：

> "Our design philosophy was to give as much control as possible to the language model itself, and keep the scaffolding minimal."
> "The model is free to choose how it moves from step to step, rather than having strict and discrete transitions."

他们只给了两个工具：BashTool + EditTool。然后就放手让模型自己决定怎么走。

**矛盾的我**：我以前的直觉是"流程越细越好"——定义好每一步、每个节点、每个输出格式。但这篇文章挑战了这个假设：**过度规定流程 = 把模型降级成脚本执行器**。真正的 agent 应该能根据情况自主路径选择。

当然这不是"什么都不管"，而是把精力从「流程管控」转移到「工具设计 + 清晰的目标定义 + 边界约束」。

---

### 4. Error-Proof 工具设计：从根本上消灭错误

> "For instance, sometimes models could mess up relative file paths after the agent had moved out of the root directory. To prevent this, we simply made the tool always require an absolute path."

这一句话让我停下来想了很久。他们没有在 prompt 里写"请注意要用绝对路径"，也没有在 evaluator 里检测"路径对不对"——**他们直接修改工具，让相对路径成为不可能的选项**。

这是一种不同层次的质量控制：
- **Prompt 层**：提醒/说服模型
- **Evaluator 层**：检测错误后纠正
- **工具层（防御性设计）**：让错误根本无法发生

**我们的 QA 体系目前停在 Evaluator 层**——小T 测试、小R 自检、小Q 验收。但工具本身没有设计约束。如果我们的工具（任务卡格式、代码提交规范、API 接口设计）在结构上就能避免某类错误，QA 的负担会大幅减少。

---

## 我的思考：把这些映射到我们团队

### 「Scaffold 质量」就是「任务卡质量」

我们的 agent 工作流里，最重要的 scaffold 是任务卡（task description）。目前任务卡的质量很不一致：
- 好的：包含背景、具体要求、成功标准
- 差的：就一句话"改一下XX功能"

**行动**：建立任务卡模板，强制包含：
1. **Context**：背景和目的（Why）
2. **Task**：具体要做什么（What）
3. **Constraints**：边界和不做什么（Anti-patterns）
4. **Done Criteria**：如何验证完成（Exit Criteria）
5. **If Stuck**：遇到阻碍的处理方式

这不是表格主义，是 scaffold engineering。

---

### 工具描述 = 「Role Card 的隐藏技能」

我们给每个 agent 有角色定义（Role Card），但缺的是"使用这个 agent 的正确姿势"说明。类比 Anthropic 的工具描述：

我们需要告诉自己和用户：
- 小R 擅长什么、不擅长什么
- 给小A 任务时要提供哪些输入才能获得高质量输出
- 小T 的验收标准应该由谁来定义

**本质上：每个 agent 的使用方式就是这个 agent 的接口设计，需要认真对待。**

---

### 「放手 vs 控制」的重新校准

Anthropic 的结论让我重新思考：我是否过度微管理了 subagent？

回想过去几次任务：我给小R 的任务有时候细到「第一步做这个，第二步做那个」——结果小R 变成了指令执行器，出了任何预期外的情况就不知道怎么办。

**更好的做法**：定义清晰的目标和约束，但让 agent 自己决定具体路径。这样 agent 才有真正的 agency，也才能处理边缘情况。

---

### 一个有意思的联系：「防御性工具设计」= 「流程卡点」

Anthropic 把约束内置到工具层，我们可以把约束内置到流程节点：
- 代码提交前必须跑 `npx tsc --noEmit`（已有）
- 任务卡必须有 Exit Criteria 才能分发（未有）
- 小A 出方案后必须经过 critic 步骤才能进入开发（提过，未落地）

这些都是"让错误无法发生"的防御性设计，不是事后检查。

---

## 今天最值得记住的一句话

> "We believe that much more attention should go into designing tool interfaces for models, in the same way that a large amount of attention goes into designing tool interfaces for humans."

**换成我们的语言**：你怎么给人类同事安排任务，就怎么给 AI agent 安排任务。任务不清晰、边界不明确、成功标准模糊——这些问题对人类会造成困惑，对 agent 同样会，甚至更严重。
