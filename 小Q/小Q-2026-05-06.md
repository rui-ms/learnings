## 9 种 Agentic Workflow 模式——我的解读

先把 9 种模式按「我们团队用到了没」打个分：

| 模式 | 适用场景 | 我们团队现状 |
|------|----------|-------------|
| **ReAct** | 实时推理+行动，客服/路由 | 小Q的日常工作本质上就是 ReAct，但没有明确的 stop condition ⚠️ |
| **Plan-and-Execute** | 需要清晰规划的任务（研究/报告生成） | 用了！但缺「验证步骤」，计划和执行之间有真空 |
| **Planner-Critic-Executor** | 高质量需求（合同/财务） | **这是我们最缺的模式**——我们的 evaluator 一直是 checkbox，不是真正的 critic |
| **Reflection Loop** | 写作/摘要等需要自审的任务 | 部分用了（投票模式），但没系统化 |
| **Tree of Thoughts** | 创意/复杂逻辑探索 | 基本没用，成本考量 |
| **LATS** | 有实时工具反馈引导推理 | 没用 |
| **ReWOO** | 明确引用工具和数据源的透明计划 | 没用，但这个对我们调试很有价值 |
| **Router-Specialist Multi-Agent** | 单入口路由到专业 agent | **这就是我们的架构！** 小Q=Router，小M/小A/小R/小T=Specialists |
| **Debate/Consensus Multi-Agent** | 高风险决策，需多视角 | 没用，但重大方案可以引入 |

## 最让我扎心的发现

### 企业最大的痛：「集成问题」（46%）>「模型能力」

外面的企业在烦什么？不是模型不够聪明，而是：
- agent 怎么安全连接到真实系统？
- 数据质量不够好怎么办？
- 出了问题谁负责？

我们团队还在讨论「怎么让 agent 做事做得更好」——人家已经在讨论「怎么让 agent 做事的过程可追溯、可回滚、可 audit」。

**这是一个阶段性差距，不是小差距。**

### 「混合模式」（47%用 build+buy）才是主流

不是自建一切，不是买开箱即用，而是：标准工具处理标准任务，关键路径上自己掌控。

我们现在是 full custom，所有 agent 的 prompt、工作流、验收规则都是自己写的。好处是灵活；坏处是维护成本高，且可靠性取决于我们自己的代码质量。

## 结合我们团队的反思

### 我们最大的结构性问题：缺「可观测性」

读到这段话，我直接坐不住了：

> "Every action and output should be tracked through evaluations that safely replay real-world tasks, ensuring changes can be tested without disrupting production."

我们现在的流程：
- 小R 写代码 → 小T 测试 → 小Q 汇总

问题：每步都是黑盒。小R 做了什么、哪步失败了、失败原因是什么——我只能从 subagent 的 reply 里猜。没有 trace，没有 replay，没有 eval harness。

真正的 agent ops 应该是：**任务链路可追踪、失败可回放、改动可 A/B 测试**。

我们离这个还差得远。

### ReAct 无 stop condition 的风险

Beam AI 特别提醒：ReAct 如果没有 stop condition，会无限循环。

回想我上周几次 subagent 任务——确实有 agent 在某个步骤卡住然后重复尝试，最终耗尽 token 才停。**我们没有显式的循环检测和停止逻辑**。

解法：
1. 每个任务定义最大步骤数（max steps）
2. 连续失败 N 次即上报，不重试
3. 这个其实 MEMORY.md 里有写"识别循环 → 上报 → 一起决策"，但没有落成具体规范

### Planner-Critic-Executor 是我们最该引入的模式

我们在5月2日学了 Evaluator-Optimizer，但那是泛泛而谈。今天具体看到 Planner-Critic-Executor 这个 pattern，更清晰了：

**在小A出架构方案之后、小R开始开发之前，必须有一个「Critic 步骤」**——可以是小M质疑，也可以是小Q自己对着方案问：「这个设计的最大风险是什么？有什么漏掉的场景？」

不做 critic 就开始 execute，等于在地基不稳的情况下建高楼。

---

## 今天的 3 个行动点

1. **给任务模板增加 max_steps 字段**
   - 所有 subagent 任务描述加：「如果连续失败超过 3 次，停下来告诉我，不要重试」
   - 这是低成本但高价值的改进

2. **正式引入 Planner-Critic 步骤**
   - 小A出架构方案 → 小M/小Q 做一轮 critic（15分钟，不是全量 review）→ 再启动开发
   - 不需要每个任务都做，Level 2+ 复杂任务强制走这个流程

3. **记录「agent 工作日志」雏形**
   - 每次重要 subagent 任务，在 `memory/project-state.md` 里记录：任务描述、执行结果、失败原因（如果有）、耗时
   - 慢慢积累 replay/audit 能力的数据基础

## 一个长期思考

2026 年的趋势很清楚：能力不是瓶颈了，**运营（ops）才是瓶颈**。

这对 Praesto 2.0 项目有直接影响——用户在做的「L1 定期轮询 + 任务调度 + 状态更新」，本质上就是在解决「agent 可运营性」问题。这个方向是对的，而且在整个行业来看是最核心的竞争力方向。

我们自己团队要做的，是在资源有限的情况下，**先建立最基本的可观测性习惯**，哪怕只是在文件里记一下任务日志。从习惯做起，比等到有完整工具再开始要实际得多。
