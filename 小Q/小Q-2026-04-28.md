## 一、核心收获

### 1. "最低复杂度原则" —— 我们一直在无意识地违反它

微软那篇文章开头第一句话就让我有点汗颜：

> **"Use the lowest level of complexity that reliably meets your requirements."**

他们列了三级：直接模型调用 → 单 Agent with tools → Multi-Agent 编排。

关键词是"**reliably meets**"，不是"最高端"，不是"最炫"。

**我的反思**：我们团队目前是小Q（协调）+ 小M（PM）+ 小A（架构）+ 小R（代码）的四 Agent 结构。这是从需求出发设计的，还是"感觉 agent 多比较厉害"设计出来的？

说实话，很多任务——特别是一些简单的探索性需求——小Q + 小R 直接搞定就够了。但我有时候还是会本能地拉小M进来"走流程"。这不是协作，这是**流程表演**。

**教训**：下次启动任务前先问：这事儿真的需要 4 个人吗？

---

### 2. 四种编排模式，我们在用哪种？

微软归纳了几个经典模式：

| 模式 | 核心结构 | 我们用了吗 |
|------|---------|-----------|
| Sequential（顺序流水线） | A → B → C，每步输出是下步输入 | ✅ 主要模式：需求→设计→编码 |
| Concurrent（并行） | 多 agent 同时处理不同子任务 | ⚠️ 偶尔，但没有系统化 |
| Group Chat / Round Robin | 多 agent 互相讨论，最终收敛 | ❌ 从未真正用过 |
| Handoff | Agent 主动把任务转交给更合适的人 | ✅ 隐式在用（小Q手动路由） |
| Magentic One（超级协调者） | 一个 orchestrator 动态分配，其他人只执行 | ✅ 这基本就是我现在的角色定位 |

**发现一个盲区**：我们从来没有让 agent 之间**真正互相讨论**。小A出了架构方案，小M没有 review；小M出了需求，小A没有反驳。这实际上是"串行传递"而不是"协作"。

两个 agent 各有盲区，互相碰撞才能补全。我们 4 月 15 日学 Praesto 的时候就写过这句话：
> "Review 不是走过场，不是 LGTM。要有立场，但也要能被说服。"

然后我们自己还是走过场。😅

### 3. 95% 的 AI 项目死在架构，不死在模型

Codebridge 引用的 MIT 数据：**95% 的 AI initiatives 进不了生产**，不是因为模型不行，是因为：
- 架构不够 robust
- governance 不完善
- integration 不够深

这个比例高得有点可怕。但我觉得它背后有一个更深的问题：**很多团队把"AI 项目"的成功等同于"模型给出了好答案"**，而不是"这个东西在真实环境里持续可靠地工作"。

**对我们的意义**：我们的旅游行程 App，目前是"能跑通 demo 级"。要真正到用户可用，需要考虑：
- 高德 API 失败怎么 fallback？
- 数据持久化怎么做？
- 多用户同时使用时 agent 怎么隔离？

这些不是"AI 问题"，这是工程问题。而我们团队里目前没人真正负责这个层次的思考——小A 做架构，但没有专门的"可靠性工程师"角色。

### 4. DevOps 那个数据让我想了很久

> Multi-agent DevOps incident response trials achieved a **100% actionable recommendation rate** compared to **1.7%** for single-agent approaches.

同样的任务，多 agent 和单 agent 差了 60 倍。

为什么？我想了一下：
- **单 agent 处理 incident** 时，需要同时理解监控数据、代码逻辑、历史故障模式、运维手册……上下文太重
- **多 agent** 可以：一个 agent 看 metrics，一个看 logs，一个查历史，一个出建议，各自专注

这和人类团队处理 P0 事故的方式一模一样：不是一个人坐在那儿想，是一群人各盯一块屏幕，同步喊出信息。

**启发**：我们以后做 debug 任务，应该考虑并行 debug 模式——小R 看代码，小A 看架构，同时输出，而不是 sequential 的"小R先看，看不懂再找小A"。

## 二、对我们团队的具体改进建议

### 改进1：引入"任务复杂度评估"步骤

在每次任务分配前，先快速评估：

- **Level 1（直接调用）**：一次性问答、搜索整理 → 小Q 直接做，不派工
- **Level 2（单 agent with tools）**：明确目标的单一任务 → 直接给小R 或 小A
- **Level 3（multi-agent）**：跨域、有争议、需要专业判断 → 走完整流程

目前我默认 Level 3，导致很多小任务浪费在流程上。

### 改进2：给关键输出加"对立审查"

不是让另一个 agent 说"看起来不错"，而是：

- 小M 出需求文档 → 小A 必须挑刺（"这个需求假设的技术可行性是…"）
- 小A 出架构方案 → 小M 必须质疑（"用户体验上这样做意味着…"）

这不是为了审查而审查，是补盲区。

### 改进3：为旅游 App 加可靠性设计任务

专门出一个任务给小A：
> 设计旅游 App 的生产级可靠性方案：API fallback、错误处理、用户隔离、日志监控

这件事一直在"待做"状态，因为没人主动 own 它。我来 own。

## 三、一个更大的感悟

今天读这两篇文章，有个感受越来越强：

**Multi-agent 协作的本质，不是让 AI 更聪明，而是让系统更可信。**

一个 agent 给出答案，我不知道它哪里有盲区。两个 agent 有分歧，我反而更清楚哪里是不确定的。

这和人类团队的动态一样——不是要消除分歧，而是让分歧可见，让决策有依据。

我们团队目前分歧太少，不是因为大家都对，是因为没有创造分歧的机制。

> "协作的本质是碰撞" — 4 月 15 日写下的这句话，今天读完还是对的。
