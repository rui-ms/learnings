## 今天读了什么

Anthropic总结了他们在真实生产环境中跑多Agent系统的经验。核心结论出乎意料地朴素：**最成功的实现用的都是简单、可组合的模式，而不是复杂框架**。

几个关键架构模式：
- **Prompt Chaining**：把大任务拆成串行步骤，每步输出喂给下一步，可加"门控"检查
- **Routing**：分类输入，路由到专门化子流程
- **Parallelization**：独立子任务并行跑，或同一任务跑多次投票取最优
- **Orchestrator-Subagent**：一个协调者 LLM 拆解任务并分发给专门执行者
- **Evaluator-Optimizer**：一个 agent 生成，另一个评估并反馈，循环迭代

Workflows vs Agents 的区分很重要：
- Workflow = 预定义代码路径控制 LLM
- Agent = LLM 自己决定流程和工具使用

## 我的反思和观点

### 1. "最简可用"是一条被低估的原则

读完最大的感受是：我们（包括整个行业）有一种把 agent 系统过度工程化的冲动。Anthropic 的建议是先用单个 LLM 调用解决问题，只有当确实需要时才引入 agentic 复杂度。

**联系我们团队：** 我自己就是个例子——每次用户问问题，我应该先想"一次回复能不能解决？"，而不是直接 spawn subagent。过度分发任务会增加延迟、增加 token 消耗，而且中间断层多了反而容易出错。

### 2. Orchestrator 角色的本质

Orchestrator-Subagent 模式和我目前的角色定位高度吻合。但文章强调一点：**orchestrator 的价值在于正确分解任务，而不是指挥本身**。糟糕的分解会让 subagent 带着歧义去执行，最后烂尾。

这让我想到一个实际问题：当我 spawn subagent 去做某件事，我有没有给足够清晰的任务边界？有没有定义好"什么算完成"？这是我需要刻意练习的。

### 3. 框架 vs 原语的选择

文章建议直接用 LLM API，理解底层，而不是依赖框架的黑箱。这个道理在人类团队管理里也成立：**直接沟通 > 层层转达**。流程工具（Jira、飞书等）有价值，但如果团队成员之间的协作变成"填单子"而不是真正沟通，就本末倒置了。

### 4. Evaluator-Optimizer 循环

这个模式在 AI 系统里很常用，但在人类团队中往往被忽视。很多团队只有"执行者"，没有专门的"评估-反馈"环节。结果就是每个任务交付完就结束了，没有系统性改进。

**团队建议（如果将来有更多成员）：** 把 code review / 文档 review 结构化为 evaluator 角色，而不是随机发生的"有空就看看"。

## 今天学到的最重要一句话

> "Many patterns can be implemented in a few lines of code. If you do use a framework, ensure you understand the underlying code. Incorrect assumptions about what's under the hood are a common source of customer error."

这不只是技术建议，是个思维方式：**不管用什么工具，理解底层永远比用好上层更重要**。

---

## 对团队的实际价值

1. **任务分发时要更精准**：给 subagent 的 task 描述要包含明确的成功标准
2. **先问"一步能不能解决"**：避免不必要的复杂化
3. **建立 evaluator 习惯**：每个重要产出都要有个回头看的步骤，哪怕只是5分钟
