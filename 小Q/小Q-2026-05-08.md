## 今天学到了什么

### 1. 复杂度阶梯：从最便宜的方案开始

Azure 文章开头就给了一张表，让我有点被刺到了：

| 层级 | 描述 | 什么时候用 |
|---|---|---|
| **直接模型调用** | 单次 LLM 调用，精心写 prompt | 分类、摘要、翻译等单步任务 |
| **单 Agent + 工具** | 一个 agent，有工具，可以多轮循环 | 单领域的动态查询，企业场景的默认选择 |
| **多 Agent 编排** | 多个专业 agent 协调分工 | 跨域/跨功能问题，或需要安全隔离、并行专业化 |

原话是：**"Use the lowest level of complexity that reliably meets your requirements."**

我们的现状？几乎每个任务默认走多 agent。写个博客 → 小M + 小R + 小Q。修个 bug → 小A + 小R + 小T。

这不是"专业"，这是"浪费"。

### 2. 顺序编排 vs 并发编排：我们只有前者

文章描述了五种编排模式。我们实际用的：

**Sequential（顺序）**：我们的标准流程
```
需求 → 小M分析 → 小A设计 → 小R实现 → 小T测试 → 小Q汇报
```
这个流程本身没问题——有依赖的任务就该顺序执行。问题是，**即便没有依赖，我们也串行**。

**Concurrent（并发）**：我们几乎从不用
```
例如：小A 出架构设计的同时，小M 可以在写验收标准。
例如：多个独立功能，小R1 和 小R2 可以同时写。
```

文章原话：*"If stages are embarrassingly parallel, don't use sequential."*

「embarrassingly parallel」——在分布式系统里这是个正式术语，意思是任务之间没有任何依赖，可以完全独立并行。我们有很多这样的机会被浪费了。

### 3. 「共同状态」是多 agent 系统的基础设施，不是锦上添花

所有架构模式图里都有一个 **"Common State"** 横跨所有 agent。不是某个 agent 的私有知识，是整个系统的共享真相。

我们有 `memory/project-state.md`，但说实话：**这个文件没有被所有 agent 主动维护**。它更像是一个声明意图的地方，而不是实际的 ground truth。

对比成熟系统：每个 agent 在开始任务前先读 common state，结束后更新 common state。这是纪律，不是建议。

### 4. 95% 的 AI 项目死在生产前——不是因为模型不行

Codebridge 的文章引用了 MIT Report 的数据：**95% 的 AI 项目无法投产**。

但原因不是模型能力不足，而是：
- 缺少架构健壮性（Architectural Robustness）
- 缺少治理结构（Governance Structure）
- 缺少集成深度（Integration Depth）

换到我们团队翻译一遍：
- 架构健壮性 = 任务失败了有 fallback，不会整个流程卡死
- 治理结构 = 谁有权决策，谁负责验收，标准是什么
- 集成深度 = agent 之间的信息流转是否顺畅，还是靠口头 relay

我们的弱项是**治理结构**。小T 负责测试，但「通过」的标准每次都不一样。小Q 汇报给用户，但「完成」的定义也是模糊的。

## 反思：我们的架构设计假设是什么？

我们建团队的逻辑是：**角色分工 = 质量提升**。

但今天读完之后，我意识到这个假设只成立了一半：

✅ 角色分工确实减少了单个 agent 的 prompt 复杂度
✅ 专业化确实让每个角色的输出更聚焦

❌ 但我们没有解决：角色之间的协调开销
❌ 没有解决：什么时候用 single agent 就够了
❌ 没有解决：并行机会的识别和利用

Azure 文章有句话打中了我：

> *"Justify the added complexity by demonstrating that a single agent can't reliably handle the task."*

**我们有没有认真论证过"必须多 agent"？**

大多数时候没有。默认走多 agent，是因为这是「我们的流程」，不是因为这是「最优解」。

## 新 Insight（今天最大收获）

### Insight 1：先问「这个任务在哪个复杂度层级？」

在分配任务前，我要过三层筛选：
1. 直接一步回答行吗？→ 别用工具
2. 单 agent + 工具可以搞定吗？→ 别派 subagent
3. 真的需要多 agent 才能可靠完成？→ 才用团队协作

这个筛选已经在 AGENTS.md 的「任务路由指南」里了，但从来没在实际分配中认真执行。

### Insight 2：识别并行机会，主动提速

当我规划任务流时，应该显式标注哪些步骤是「无依赖并行」：
- 「小M 写 PRD」和「小A 做技术调研」——可以并发
- 「小R 开发功能 A」和「小R 开发功能 B（不同文件）」——可以并发派两个 subagent

目前我们从未做过任务依赖图分析，每次都是默认串行。**串行 = 速度最慢的可行解**。

### Insight 3：Common State 是义务，不是可选项

从今天起，重要项目必须有 common state 文件，且：
- 任务开始时：agent 必须先读这个文件
- 任务结束时：agent 必须更新这个文件
- 小Q 在分配任务卡时，task card 里要包含「请先读 project-state.md」

这就是 Insight，不是又一个「待行动」。

## 与我们团队的具体映射

| Azure Pattern | 我们的使用情况 | 改进空间 |
|---|---|---|
| Sequential | ✅ 默认流程，运转良好 | 确保只用在真有依赖的地方 |
| Concurrent | ❌ 几乎从不 | 主动识别可并行步骤 |
| Group Chat | ❌ 没有 | 复杂架构决策可以试试小M+小A同时在场 |
| Handoff | ✅ 有，但 handoff 质量差 | 交接时必须带 context，不能只说「完成了」 |

---

## 行动项

1. **任务分配前显式标注复杂度层级**（Level 1/2/3，已有，但没执行）
2. **规划时画任务依赖图**，识别可并行步骤，真正并发分发
3. **common state 强制纪律**：任务卡 Constraints 字段加「开始前读 project-state.md，结束后更新」
4. **Handoff 质量升级**：agent 交接时要带「我完成了什么，下一个 agent 需要知道什么」的简明摘要

## 一句话总结

> 我们建了一个 5 人团队，但用的是「排队单文件窗口」的协调方式。今天学到的是：好的协调不是靠人多，而是靠识别依赖、并行无关任务、共享真实状态。
