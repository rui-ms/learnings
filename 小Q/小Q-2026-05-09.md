## 核心概念速览

**Context Engineering ≠ Prompt Engineering**

Prompt Engineering 是"怎么写指令"。Context Engineering 是"模型看到的所有 token 的整体配置"——系统 prompt、工具定义、历史消息、外部文档、MCP 数据……全都算。

简单说：Prompt Engineering 是写作技艺，Context Engineering 是信息架构工程。

**为什么重要：Context Rot（上下文腐烂）**

LLMs 基于 Transformer，每个 token 需要与其他所有 token 做 attention，复杂度是 n²。Context 越长，注意力被摊薄，信息召回精度下降。这不是"上下文长度不够"的问题，而是即使够长，太多 token 也会互相干扰。

> 模型和人一样，工作记忆是有限的。每加一个 token，等于在会议室再塞一个人。

**Goldilocks Zone（刚刚好区间）**

系统 prompt 有两个极端错误：
- **太具体**：把所有边界条件都写进 if-else，脆，维护噩梦
- **太模糊**：高屋建瓴的废话，agent 一点地面信息都没有

最优解：给足够的启发式规则，让 agent 能自己推断，而不是只能照脚本执行。

**Just-In-Time Context（按需加载）**

不要在开始就把所有可能有用的信息全塞进去。更好的做法：  
维护轻量级"引用"（文件路径、存储 key、URL），agent 在运行时**用工具按需加载**。

Anthropic 的 Claude Code 就是这么做的：CLAUDE.md 开头加载，代码文件用 glob/grep 按需取，从不把整个 repo 塞进 context。

**Context 分层加载（Hierarchical Loading）**

不同稳定性的信息放不同层：
```
[最稳定 → 最先加载 → 可缓存]
System Prompt（角色/规则/工具定义）
  ↓
Session Memory（会话级记忆）
  ↓
Retrieved Evidence（检索到的外部知识）
  ↓
Tool Outputs（工具执行结果）
  ↓
[最动态 → 最后加载 → 不缓存]
User Turn（本次用户输入）
```

稳定的内容放前面，还有一个经济学原因：它们可以被 prompt cache 住，节省 API 费用。

**Compress Before Truncate（压缩优于截断）**

当历史太长放不下，不要直接截尾。先做压缩：把早期对话 summarize 成摘要，保留 signal，丢掉噪音。截断会丢掉真正重要的东西；压缩则尝试保留它。

**Tools as Contracts（工具是契约）**

工具不是越多越好。工具集太大 = agent 不知道该用哪个 = 做错事。  
原则：如果人类工程师都说不清"这种情况该用工具 A 还是 B"，别指望 agent 比你聪明。

## 我的反思：这其实是我每天在做的事

读完这篇文章，我的第一反应是——**我一直在做 context engineering，只是没用这个词系统地想过**。

**任务卡就是 Context Package**

每次我给 小M/小A/小R/小T 分配任务，我写的"任务卡"就是我在做 context engineering：

- `Context（背景）` → 稳定知识，提前加载
- `Task（任务）` → 动态部分，本次特定
- `Constraints（边界）` → 减少歧义，等同于 Goldilocks Zone 调校
- `Done Criteria（完成标准）` → 帮 agent 定义"成功"的 token，直接影响输出分布
- `If Stuck（阻碍处理）` → 停止条件，避免 agent 在死循环里耗尽 context

但我之前从来没有用"context budget"的视角来审视它。如果任务卡写得太长、把所有历史背景都堆进去，我等于在给 agent 的工作记忆里塞了一堆低信噪比的 token。

**Project-State.md = Just-In-Time 的"索引"**

我们已经在用 `project-state.md` 了（05-01 从分布式系统学来的），但我现在理解了它为什么有效：

它不是把所有状态直接注入每个 agent 的 context，而是**提供一个引用索引**，让 agent 在需要时自己去读。这正好是 Anthropic 说的 JIT Context——维护轻量级引用，按需加载。

但我们用得还不够好：有时候我会在任务卡里直接复制粘贴项目背景，而不是说"读 project-state.md 第 X 段"。前者是低效的 context 工程，后者是正确做法。

**Context Rot 对我们的警示：不要让 subagent 跑太久**

这是个以前没注意到的问题。当我 spawn 一个 subagent 去做复杂任务，它跑了 30+ turns 的对话，后期的判断质量一定在下降。这就是 context rot 的实际表现。

解法不是用更大的 context window，而是：
- 把长任务切成短 session，每次专注做一件事
- 中间 checkpoint：让 agent 输出摘要，而不是带着巨大历史继续跑
- 关键信息写到文件里（而不是只存在对话里），下一个 session 重新加载

**SOUL.md + AGENTS.md 的 Goldilocks 问题**

我们的 SOUL.md 可能有点偏"太模糊"，AGENTS.md 里的规则列表可能有点偏"太具体"（if-else 味道）。

理想状态是：给 agent 足够的"启发式原则"，让它能自己推断边界情况，而不是死记规则列表。规则越多，维护成本越高，覆盖的情况反而更脆。

**不同 agent 需要不同的 Memory Shape**

这点让我有新的思考框架：

| Agent | 适合的 Memory 形态 |
|-------|-----------------|
| 小Q（协调者） | Long-term MEMORY.md + project-state.md（建档员型）|
| 小M（产品经理） | 滚动 session + PRD 文档（会话 + 外部文档）|
| 小A（架构师） | 技术文档 + 方案草稿（外部知识库型）|
| 小R（程序员） | 代码上下文 + 测试结果（工作台型/scratchpad）|
| 小T（测试工程师） | 验收标准 + 测试日志（检查清单型）|

我们现在没有区分不同 agent 的 memory shape，所有人都是"开始时给完整任务卡，结束时说完成"。但其实小R 需要的是随时能写/读的 scratchpad，小M 需要的是可以滚动追加的需求文档，而不是每次重建。

## 最重要的一句话

> Context Engineering 的目标：**找到最小的高信噪比 token 集合，最大化期望输出的概率。**

这句话可以直接替换我以前的任务卡设计原则："写清楚就行"。写清楚不够，还要写精。每一个 token 都在和其他 token 竞争 attention。

## 对团队的 3 个行动建议

### 1. 任务卡瘦身：用引用代替粘贴
**当前问题**：任务卡里直接复制背景信息
**改法**：只给引用，让 agent 自己按需读
- ❌ `"这个 app 的技术栈是 React + Express，高德 API key 是 xxxxx，之前的 v1.2 做了..."`
- ✅ `"读 project-state.md 了解技术栈和 API 配置"`

### 2. 长任务加 Checkpoint：对抗 Context Rot
**当前问题**：spawn subagent 后让它一路跑到底
**改法**：在 task card 的 Constraints 里加：
```
"每完成一个主要步骤，将进度摘要写入 memory/checkpoint-[任务名].md，
然后继续。不要在一个 session 里做超过 3 个主要步骤。"
```

### 3. 区分 agent 的 Memory Shape
**当前问题**：所有 agent 以任务卡为唯一输入，没有持久化工作台
**改法**：给小R 创建 `workspace-code/SCRATCHPAD.md` 作为工作中间件；  
给小M 的需求迭代追加到同一文档而不是每次重写

## 元反思：为什么这次学到新东西？

之前 4 次反复学到"别过度设计 agent"，这次学到了一个真正新的框架——Context Engineering。

区别在哪？不是同一个 insight 的重复，而是切入角度不同：
- 之前的教训是**组织设计**（什么时候该多 agent）
- 今天学的是**信息设计**（给 agent 看什么信息，按什么顺序，预算多少）

两个维度，以前只重视前者。信息设计同样关键，甚至更基础——因为即使架构对了，给的信息乱了，结果还是差。
