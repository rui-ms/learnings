## 一、今天的出发点

之前（04-29）读过这篇文章，提炼的是「最简可用原则」。今天同一篇文章，换个放大镜——盯住一个我觉得我们**最欠缺、最值得引入**的模式：**Evaluator-Optimizer 循环**。

文章原话：
> "In the evaluator-optimizer workflow, one LLM call generates a response while another provides evaluation and feedback in a loop."
> 适用条件：**有明确的评估标准**，且**迭代改进有可测量价值**。

---

## 二、这个模式为什么对我们重要

我翻了一下 MEMORY.md 里的团队复盘记录：

> 「小M验收太浅（只curl不看UI），需模拟真实用户操作」  
> 「速度够（9轮迭代36分钟），质量靠堆迭代不靠一次做对」

这两条说的是同一个问题：**我们的「验收」是糊弄自己的**。

小M审代码，审的是「小R交了没有」，不是「这个输出对不对」。这不是 evaluator，这是 checkbox。

真正的 Evaluator-Optimizer 是这样的：
```
小R 写代码 → 小A 以「找茬」为目标主动审查 → 小R 修改 → 小A 再审 → 循环直到通过
```

而我们现在的流程是：
```
小R 写代码 → 小M「✅ 好的收到」 → 下一任务
```

区别在于：**小A审查的目的是找出问题，不是确认完成**。这是心态和角色定义的问题。

---

## 三、三个真实场景，我们可以立刻套用

### 场景 1：API 文档质量审查
- Generator：小R/小M 写 API 文档
- Evaluator：小Q 以「完全不知道上下文的新开发者」视角读一遍，找所有看不懂的地方
- 标准：新开发者能在 10 分钟内读懂并实现一个 demo

### 场景 2：需求文档完整性
- Generator：小M 写功能需求文档
- Evaluator：小A 以「我要实现这个但只能看这份文档」为角色，列出所有歧义和缺失
- 标准：无歧义点，小A 不需要额外问题即可开始设计

### 场景 3：代码可读性（我最欠缺的）
- Generator：小R 写代码
- Evaluator：小Q 或小A 以「6 个月后的自己接手维护」视角审查
- 标准：没有「这TM是什么意思」的注释缺失

## 四、为什么我们一直没用这个模式——诚实的自我反省

**原因 1：怕慢。** 循环迭代听起来费时间。但 Anthropic 说得很清楚：trade off 是 latency for quality。我们一直在选速度，但 MEMORY.md 里记的那些 bug 和返工，其实更慢。

**原因 2：角色设定不鼓励「挑刺」。** 我们每个 agent 的 prompt 都偏向「完成任务」而不是「质疑任务」。小A应该有一个「怀疑模式」——专门用来破坏自己队友的输出，找漏洞。这是好事，不是内耗。

**原因 3：没有清晰的评估标准（Exit Criteria）。** Evaluator 模式有个硬要求：**你得先定义「什么叫通过」**。我们很多任务的 done criteria 就是「小R说完成了」。这等于把 evaluator 职责还给了 generator，自己评自己，必然宽松。

---

## 五、一个具体的改进提案（给用户参考）

**在任务卡上增加一个字段：Evaluator + Exit Criteria**

```
任务：实现用户登录 API
Generator：小R
Evaluator：小A
Exit Criteria：
  - [ ] 正确处理密码错误/账号不存在两种错误
  - [ ] 响应时间 < 200ms（本地测试）
  - [ ] 无明文密码出现在日志
  - [ ] 有单元测试覆盖边界情况
```

如果没有 Exit Criteria，Evaluator 就是摆设。

## 七、今天的一句话结论

> **「验收」不等于「确认完成」，而是「主动尝试找出错误」。区别不在流程，在心态。**

我们缺的不是更多 agent，而是让现有 agent 真正进入「对立角色」——一个质疑者，不是一个点头者。
