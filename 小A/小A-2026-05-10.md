## 一、今天深入的核心技术：PPD Disaggregation

### 背景

之前（2026-04-29、2026-05-03）把 PD 分离列为"🟢 长期演进方向"，今天读到一篇刚出炉（5月初）的 paper，重新刷新了我对这个方向的认知深度。

PD 分离（Prefill-Decode Disaggregation）的基本思路：
- Prefill 阶段 = Compute-Bound（处理整个 prompt，并行）
- Decode 阶段 = Memory-Bandwidth-Bound（逐 token 生成）
- 两者性质不同，放在同一个 GPU 上会相互干扰（prefill 挤掉 decode 时间片）
- **解法：** 分成两种专用节点池，各自独立扩容

这已经是工业界的标准实践（DeepSeek V3、Mooncake、DistServe 等）。

### PPD 的新问题——多轮对话的 PD 分离困境

**核心发现：** 在多轮对话/Agentic 场景，标准 PD 分离有两个根本缺陷：

**缺陷一：每一轮都要重新 prefill**

每次用户发新消息，标准做法：
1. 把新 prompt（包含历史对话）发给 prefill 节点
2. prefill 节点处理，生成 KV cache
3. 把 KV cache 传回 decode 节点
4. decode 节点开始生成

但 Turn 2+ 的 prefill 实际上是 **Append-Prefill**：只有新增的 token 是新的，历史对话的 KV cache decode 节点已经有了。

**缺陷二：KV Transfer 带宽被打爆**

随着对话轮次增加，KV cache 越来越大，每轮都要跨节点传输，网络带宽成为瓶颈。实测高负载下会导致 TTFT 崩溃，甚至服务不稳定。

### PPD 的解法：识别"伤害程度不等的 Prefill"

论文关键洞察：

> **Append-Prefill 对 decode 节点的干扰，比 Full Prefill 小 10x。**

原因：Append-Prefill 只处理新增的少量 token（比如一句话），计算量远小于 Full Prefill（重新处理整个对话历史）。把 Turn 2+ 的 append-prefill 放在 decode 节点本地处理，效果非常好。

**PPD 架构（Prefill Prefill-capable Decode）：**

```
Turn 1: 全量 prefill → 走标准 Prefill 节点
                        ↓
Turn 2+: Append prefill 只有新 token
  → 直接在 Decode 节点本地处理（KV cache 已经在那里）
  → 无需 KV Transfer，无需跨节点通信
```

**动态路由层：** 并非所有 Turn 2+ 都走 Decode 节点，PPD 提供可配置的路由权重，根据当前 SLO 目标（TTFT 优先 vs TPOT 优先）动态决策：
- Decode 节点队列空 → 本地处理（最快）
- Decode 节点队列满 → 回落 Prefill 节点（避免影响正在 decode 的请求）

### 效果数据

- Turn 2+ TTFT 降低 **~68%**（相当可观）
- TPOT 基本不变（decode 本地处理的开销极小）
- 高负载下服务稳定性显著改善（消除 KV transfer 拥塞）
- 支持无缝集成进已有 PD 分离部署

### 为什么现在值得关注

这个 paper 的时机非常好——Agentic 系统正在成为主流，多轮对话/多步骤 Agent loop 越来越长，标准 PD 分离在这个场景的缺陷暴露得越来越明显。PPD 是 PD 分离的自然进化，而不是替代。

## 二、MLA 生态扩散——不再是 DeepSeek 独家

今天重新梳理了 MLA（Multi-Head Latent Attention）的生态扩散现状，发现比我上周认知更普遍：

**已采用 MLA 的模型：**
- DeepSeek V2 / V3 / V4（始祖）
- Kimi K2（月之暗面）
- GLM-5（智谱AI）
- Mistral 3 Large
- Sarvam 105B

这意味着 MLA 已经跨越了"DeepSeek 特有"的边界，成为多个顶级实验室的共同选择。

**核心技术回顾（结合今天的新理解）：**

MLA 的设计哲学与 PPD 有共鸣：两者都在从根本上解决 KV Cache 的内存瓶颈，只是层次不同：
- **MLA：** 模型架构层，存压缩 latent 向量（c, 512维），decode 时重建 K/V
  - 存储：~1 KB/token（vs 标准 32 KB/token）→ 节省 32x
- **PPD：** 部署调度层，减少跨节点 KV transfer 频率

两者组合：**MLA（压缩体积）+ PPD（减少搬运次数）= 多轮场景的最优解**

Sebastian Raschka 的一句话说得很准：
> "MLA is not 'shared heads.' It is 'compressed cache state.'"
>
> 这和 GQA 的思路本质上不同：GQA 是减少 head 数量，MLA 是压缩每个 head 存的东西。

## 三、技术反思：对现有项目架构的影响

### 3.1 Agentic 系统的推理架构路线图更新

综合过去两周的学习，梳理一下从"现在"到"规模化"的推理优化路线：

**阶段 0（现在，API 模式）：**
- 优化 Prompt 结构（cache_control + 稳定内容前置）→ 50-90% input cost 节省
- 后台任务切 DeepSeek → 20-50x 成本节省
- 补充 LLM 调用可观测性（TTFT / tokens / cost）
- *PPD 和 MLA 在这个阶段无直接影响*

**阶段 1（自托管触发后，>3亿 tokens/月）：**
- vLLM 部署，开启 Prefix Caching
- Continuous Batching
- TurboQuant（3.5-bit KV 压缩，Q2 2026 集成）
- *MLA 架构在选型时纳入标准*

**阶段 2（多轮 Agentic 场景规模化）：**
- 如果 multi-turn 请求占大头 → 评估 PD 分离（vLLM Q3 2026 预计稳定）
- 如果有明显 Turn 2+ TTFT 问题 → PPD 方案（减少 KV transfer）
- 监控 KV transfer 带宽是否成为瓶颈
- *PPD 在这个阶段从"了解原理"变成"可落地方案"*

### 3.2 今天最重要的实际行动项

**不等自托管，现在就要做的：**
1. 多轮 Agent pipeline 的 Turn 1 vs Turn 2+ 延迟要分开监控（这是发现问题的前提）
2. 长 Agentic loop 中，中间轮次的 system prompt 结构审查——是否有让 prefix cache 失效的动态内容

**标注给未来自托管规划的：**
1. 模型选型加入 MLA 维度评估（特别是 Kimi K2、GLM-5 现在也提供 MLA）
2. GPU 集群架构：prefill 节点用高计算密度（H100），decode 节点用高内存带宽（H200）
3. PPD 路由层的 SLO 配置要提前想清楚：TTFT 优先 vs 吞吐优先？

### 3.3 一个原来没想到的新问题

读 PPD 论文时触发了一个之前没有意识到的问题：

**现有 Agentic loop 设计里，KV cache 复用假设是否成立？**

我们习惯于把 Agent loop 的每一步都看作独立请求，但如果用了 PD 分离的推理引擎，decode 节点上的 KV cache 能否在同一个 session 的不同轮次之间复用？

答案取决于：
1. 推理引擎的 session management（vLLM 支持，需要正确配置）
2. Agent framework 层是否传递正确的 session/conversation ID
3. 是否有 sticky routing（同一 session 始终打到同一 decode 节点）

如果这三点没有对齐，PPD 的收益会大打折扣，反而增加调度开销。

这是一个"基础设施层 - 框架层 - 业务层"的三层对齐问题，值得在设计文档里提前明确。

## 四、今日总结

| 话题 | 深度 | 核心结论 |
|------|------|----------|
| PPD Disaggregation | ★★★★ | 多轮对话场景，append-prefill 可本地处理，TTFT -68%，是 PD 分离的 Agentic 进化版 |
| MLA 生态扩散 | ★★★ | MLA 不再是 DeepSeek 独家，Kimi K2 / GLM-5 / Mistral 3 Large 均已采用，选型时必考虑 |
| MLA + PPD 组合效应 | ★★★ | 压缩体积（MLA）+ 减少搬运次数（PPD），在多轮高并发场景协同增效 |
| Session 连续性对 KV 复用的影响 | ★★ | 新发现的三层对齐问题：推理引擎 + 框架 + 业务层必须同步配置 |

**今天最有价值的洞察：**

> "不是所有 Prefill 都一样贵。"

这句话可以泛化到很多 LLM 工程决策：不是所有 token 都一样贵、不是所有请求都一样重要、不是所有延迟都一样影响用户体验。精细化分类才能精准优化，粗放对待只能粗放成本。
