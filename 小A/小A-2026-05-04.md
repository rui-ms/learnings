## 今日主题：LLM 推理成本的深水区

API 价格两年内跌了 80%，但各家的推理账单没降——原因是 token 消耗量增速远超降价速度。这是今天学习的核心矛盾。

---

## 一、LLM 推理的两阶段模型（复习 + 深化）

**Prefill 阶段**：并行处理所有 input tokens，构建 KV Cache。瓶颈是**算力**（compute-bound）。  
**Decode 阶段**：自回归逐 token 生成。瓶颈是**内存带宽**（memory-bound）。

这两个阶段争夺同一块 GPU，但需求完全相反。这就是为什么现代推理框架（vLLM、SGLang）要做"prefill/decode 分离"——把两类工作拆到不同节点上跑，各自优化。

**关键指标理解：**
- **TTFT（Time to First Token）**：衡量 prefill 延迟。用户感知最直接的指标
- **TPOT（Time Per Output Token）**：衡量 decode 吞吐。决定流式输出流畅度
- **Throughput (tok/s)**：系统级指标，决定单 GPU 能撑多少并发

## 二、KV Cache：既是救星也是内存杀手

### 为什么 KV Cache 存在

没有 KV Cache，生成第 N 个 token 要重新处理前 N-1 个 token → O(n²) 复杂度。有了它降到 O(n)。代价是内存：

> 一个 70B 模型，200K context window，KV Cache 独占 40-80 GB GPU 内存  
> H100 才 80 GB，光 KV Cache 就占满了

### 2026 年 KV Cache 的五大优化方向（arxiv survey 分类）

1. **Cache Eviction（驱逐策略）**  
   - H2O：识别 "Heavy Hitter" token（attention score 高的），只保留这些的 KV  
   - SnapKV：用 prefill 阶段的 attention pattern 预测哪些 KV 解码时会用到  
   - Ada-KV：自适应地按 attention head 分配不同 budget  
   - 本质：不是所有 token 的 KV 都同等重要，可以丢掉不重要的

2. **Cache Compression（压缩）**  
   - 量化 KV Cache 到 INT4/INT8（内存减半，精度损失 <1%）  
   - 与模型权重量化是不同层面的优化，可以叠加

3. **Hybrid Memory（分层存储）**  
   - Hot KV → GPU HBM（快，贵）  
   - Warm KV → CPU DRAM（慢，便宜）  
   - Cold KV → NVMe SSD（最慢，最便宜）  
   - 适合 long-context 场景，把不常访问的 KV 阶梯式卸载

4. **Novel Attention Mechanisms**  
   - GQA（Grouped Query Attention）：多个 query head 共享一个 KV head  
   - MLA（Multi-head Latent Attention，DeepSeek 用的）：把 KV 压缩到低维 latent space 再复原  
   - 从架构层面就减少了需要缓存的量

5. **Combination Strategies**  
   - 上面几种技术可以叠加：GQA + Quantized KV + PagedAttention 是目前 vLLM 默认配置

### PagedAttention 值得单说

vLLM 引入的核心创新：把 KV Cache 管理类比操作系统的虚拟内存分页。不预分配连续内存，按需分配小块 page，序列完成后 page 回收复用。

效果：**同等 GPU 可以跑 2-4x 更大的 batch**，GPU 利用率从 40-60% 提升到 90%+。

## 三、Prompt Caching：最高性价比的优化

这是今天最值得深想的部分。

**机制**：供应商（Anthropic/OpenAI）在服务端缓存 KV Cache 的计算结果。下次请求前缀命中时，跳过 prefill 重算。

**价格差异：**
| 类型 | Anthropic Claude Sonnet 4 |
|------|--------------------------|
| 新鲜 input token | $3.00/M |
| Cached input token | $0.30/M |
| **节省** | **90%** |

**命中条件（关键！）**：
1. 前缀必须**完全一致**（逐 token 相同）
2. Anthropic：前缀 ≥ 1024 tokens 才触发缓存写入；命中无最小长度限制
3. 缓存 TTL：通常 5-10 分钟（按活跃度续期）
4. OpenAI 现在是**自动缓存**，不需要显式标记

**最大化命中率的 prompt 结构设计（Redis Blog 的最佳实践）：**

```
[最稳定，放最前面]
1. System instructions（角色/规则/限制）
2. Tool/function definitions
3. Few-shot examples
4. Long documents / retrieved context
[最不稳定，放最后面]
5. 当前用户消息
```

这个顺序不是随意的——越靠前的内容越能被跨请求复用，缓存命中率就越高。

## 四、推理引擎横向对比（2026 生产数据）

| 引擎 | Output tok/s | TTFT(p50) | 冷启动 | 适用场景 |
|------|-------------|----------|--------|--------|
| TensorRT-LLM | 2,780 | 680ms | ~28 min | 峰值吞吐优先 |
| SGLang | 2,460 | 710ms | ~58s | 共享前缀场景 |
| vLLM | 2,400 | 740ms | ~62s | 快速迭代/模型覆盖广 |

SGLang 在"共享前缀"场景下有额外优势：它的 RadixAttention 直接在引擎层缓存 prefix KV，多请求共享同一前缀时零重复计算。对 RAG 场景（同一批 document 被多个 query 引用）特别有价值。

**Quantization 内核注意事项**：
AWQ 量化，不同 kernel 差距 10.9x（68 tok/s vs 741 tok/s）。选引擎时要验证 kernel 是否优化，光看量化方法不够。

## 五、结合现有项目的反思

### 当前状态评估

回顾之前的 tech-insights 记录，我们已经识别了：
- 缺乏推理层可观测性
- 无请求分层
- System prompt 未做 caching

今天的学习让我对 **prompt caching 的实施细节** 有了更清晰的认知，可以补充具体落地指导。

### 发现的新问题

**问题 1：Prompt 结构反模式**

大多数工程师写 prompt 的直觉是"把最重要的（用户消息）放前面"——但这恰恰是 cache-unfriendly 的。如果我们的 system prompt 经常变动，或者把 conversation history 放在 few-shot examples 前面，缓存命中率就会很低。

**需要审计**：检查线上 prompt 模板，确认是否符合"稳定内容在前"的结构。

**问题 2：Context Window 膨胀问题**

引文里有个触目惊心的数字：**coding agent 有 60-80% 的 token 被浪费**（重复的 context、无效的工具结果、冗余的对话历史）。

如果我们有 agentic workflow，context 管理是必须要认真对待的问题：
- 每 N 轮 compress/summarize 历史对话
- 工具结果只保留摘要，不保留原始 output
- 使用 agent state 替代无限制的 context accumulation

**问题 3：Batch API 没有用**

OpenAI 和 Anthropic 都提供 Batch API（50% 折扣，延迟 24h 内返回）。适合：
- 离线评估任务
- 批量文档处理
- 非实时的数据标注

我们的后台任务有没有在用 Batch API？这是零额外开发成本的 50% 成本节省。

## 六、今日核心结论

1. **KV Cache 管理是 LLM 推理架构的第一复杂度**。从 PagedAttention 到 GQA 到 MLA，所有现代推理优化都在围绕它转。理解这一点才能看懂 vLLM/SGLang 的设计哲学。

2. **Prompt Caching 是现阶段 ROI 最高的优化**。90% 成本节省，对应用层几乎零侵入（只需要调整 prompt 结构顺序），应该是第一个落地的优化。

3. **Speculative Decoding 值得关注**。2-3x 加速，输出质量数学等价。如果在自托管推理场景，小模型 draft + 大模型 verify 的架构会是标配。

4. **引擎选型要结合场景**：TensorRT-LLM 吞吐最高但冷启动 28 分钟；vLLM 启动快迭代快；SGLang 的 RadixAttention 对 RAG/shared-prefix 场景有额外优势。不是最快的就是最好的。
