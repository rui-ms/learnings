## 核心技术笔记

### 1. 推理成本结构的本质

一个反直觉但极重要的数字：**API 价格 2024→2026 降了 80%，但总推理支出还在涨**。

原因很简单：Token 消耗量增速远超降价速度。越便宜，用得越猛。在 Agentic 系统里尤其明显——一个 coding agent 单次 loop 可能消耗 50,000 input tokens + 2,000 output tokens，团队跑 100 次就是 $18。

**成本结构拆解：**
- Output token 比 Input token 贵 3-10x（各家 API 均如此）
- 根本原因：输出是逐 token 的 autoregressive decode，输入是并行 prefill
- 结论：**减少 output token 和 cached input 的利用是降本核心**

---

### 2. KV Cache：推理内存的真正瓶颈

KV Cache 的作用是存储已生成 token 的 key-value 向量，避免重复计算注意力。没有它，生成第 N 个 token 的复杂度是 O(N²)；有了它是 O(N)。

但代价是内存：一个 Llama 3 70B 的 128K token 上下文，KV Cache 本身就吃掉约 40GB HBM。这直接限制了：
- 可服务的并发用户数
- 可支持的最大上下文长度

传统方案（PagedAttention）解决了 KV Cache **碎片化**问题（从 60-80% 浪费降到 <4%），但没有解决**总量**问题。

---

### 3. Google TurboQuant：数学优雅的压缩方案

**核心成果：** KV Cache 从 FP16（16 bit）压缩到 3.5 bit，内存节省 6x，精度零损失。

**技术路径（三步走）：**

```
原始 KV 向量 (FP16, 有outlier)
     ↓
Step 1: 随机正交旋转 (PolarQuant)
     → 能量均匀分布到所有维度
     → 消灭 outlier channel 问题（这是naive量化失败的根源）
     ↓
Step 2: Lloyd-Max 最优量化
     → 因为旋转后分布可解析（Gaussian），无需 calibration data
     → 数学最优的量化桶划分
     ↓
Step 3: QJL 误差修正（+1 bit）
     → 保证注意力分数计算的距离关系不变形
     → 防止 attention pattern 退化
```

**为什么这个方案优于 GPTQ/AWQ？**
- **Training-free**：不需要任何 fine-tuning 或 calibration 数据
- **Model-agnostic**：任何 Transformer 架构都能用
- **数学可证明**：压缩误差有 near-optimal 的理论保证，不是经验调出来的

**实测收益：**
- 4-bit TurboQuant on H100：注意力计算速度 **8x vs FP32**，约 4x vs FP16
- 3.5-bit TurboQuant：LongBench + Needle-in-a-Haystack 全面比肩 FP16
- 合并 PagedAttention + TurboQuant：等效将 GPU 的 KV 存储能力扩大 6-8x

### 4. 2026 推理优化技术栈全景

| 技术 | 解决问题 | 收益量级 | 成熟度 |
|------|---------|----------|--------|
| PagedAttention (vLLM) | KV Cache 碎片 | 2-4x 并发 | 生产就绪 |
| TurboQuant | KV Cache 内存 | 6x 节省 | ICLR 2026，社区跟进中 |
| Speculative Decoding | Decode 延迟 | 2-3x 速度 | 生产就绪 |
| Flash Attention 3 | Prefill 速度 | 2x vs FA1 | 生产就绪 |
| Continuous Batching | GPU 利用率 | 85% 成本降 | 生产就绪 |
| AWQ + Marlin Kernel | 模型权重内存 | 4x 节省 + 10.9x 速度 | 生产就绪 |
| Prompt Caching | 重复输入成本 | 90% 降低 | API 原生支持 |

**关键洞察：** Kernel 质量比量化算法本身更重要。AWQ 无优化 kernel = 68 tok/s；AWQ + Marlin kernel = 741 tok/s。**同样的权重，速度差 10.9 倍。**

### 5. Coding Agent 的推理成本特征

Coding agent 是最 inference-heavy 的场景，也是成本重灾区：
- 60-80% token 是**被浪费的**（不必要的上下文携带、冗余 tool result）
- 每个 agentic loop 的 input 远大于 output（50:1 甚至 100:1 比例常见）
- 核心优化方向：**Context Compression**（减少 input token 而非 output token）

## 对现有项目的反思与优化方向

### 🔴 立即可行：Prompt Caching 利用率提升

现有系统应该审查 system prompt 是否充分利用了各 API 的缓存机制：
- Claude：cache_control 明确标记 system prompt（cached input 价格是正常价 1/10）
- GPT：automatic prompt caching 对 ≥1024 token 的重复前缀生效
- Gemini：context caching 对超长上下文特别有效

**低挂果，做了就有即时回报。**

### 🔴 KV Cache 压缩引入路线图

TurboQuant 开源实现已在社区出现（基于 vLLM）。如果我们有自托管的 LLM 服务，应跟踪：
1. 官方 vLLM 集成 PR（预计 Q2 2026 合并）
2. 对长上下文场景（RAG 检索 + 长文档处理）的收益会最显著

### 🟡 Speculative Decoding 适用场景分析

我们的 Agent pipeline 里，有没有"decode-heavy"的步骤？典型场景：
- 长代码生成（>500 lines）
- 详细报告撰写
- 对话摘要

如果有，引入 speculative decoding 的 2-3x 速度提升值得评估。需要一个同族小模型作为 draft model（如 Llama 8B draft + Llama 70B verify）。

### 🟡 重新审视 Context 传递策略

Agentic loop 里，每一步是否都传了全量历史？很多场景只需要：
- 当前任务描述
- 最近 N 轮对话
- 精简后的工具结果摘要

**一个 20% 的 context 削减 = 20% 的成本削减，零质量损失。**

## 今日结语

从 TurboQuant 身上学到一个做架构的心法：**当经验性方案碰壁时，往往是因为前提假设错了。** Naive 量化失败，不是量化本身不行，是 outlier channel 的存在破坏了均匀性假设。TurboQuant 通过正交旋转先消灭问题的前提，才让最优量化得以发挥。

这个思路在系统设计里很通用：**与其在错误的假设上优化，不如先质疑假设本身。**
