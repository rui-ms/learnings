## 二、Speculative Decoding + FlashAttention-3：系统工程视角

### 核心思路

两者是互补的推理加速层，不是竞争关系：

- **Speculative Decoding**：解决串行解码的问题。用 Draft Model（小模型）提前生成 K 个 token，再用 Target Model 一次性并行 verify，数学上保证输出等价于纯 autoregressive，速度 2-3x。
- **FlashAttention-3**：解决每次 verify pass 本身太贵的问题。H100（Hopper）上 BF16 可达 840 TFLOPS/s，FP8 达 1.3 PFLOPS，达到理论吞吐的 ~85%。

组合收益：前者减少 verify pass 次数，后者压低每次 verify pass 的成本。两者在 **attention 占 decode 成本大头** 时效果叠加最佳。

### 2026 产线状态

这不再是 paper 级技术，已经是主流推理框架的标配：

- **vLLM**：支持 EAGLE3、MTP、PARD、MLP、N-Gram、suffix decoding 多种 speculation family；定位：中低 QPS、memory-bound workload
- **TensorRT-LLM**：低 batch 场景最优；NVIDIA 自测 3.6x 吞吐提升（H100）
- **SGLang**：多种 speculation 方法均已集成

**IBM 生产报告（值得重视）：**
- 语言模型：~2x 延迟提升
- Granite 20B 代码模型：~3x 延迟提升，同时并发用户数 ~4x

### 最关键的工程指标：Acceptance Rate

设 α 为单 token 接受率，γ 为每轮 draft 长度：
```
期望每 pass 接受 token 数 τ ≈ (1 - α^(γ+1)) / (1 - α)

α=0.5, γ=5 → τ ≈ 2.0
α=0.8, γ=5 → τ ≈ 4.1  ← 实际有意义的加速
α=0.9, γ=5 → τ ≈ 5.4
```

**α 是 speculation 能不能上生产的核心指标**。对于同分布的代码任务（N-Gram 或 EAGLE3），α 往往 >0.8；对于开放式对话，α 可能只有 0.5，speedup 有限。

### Draft Model 选型

| 方法 | 场景 | α 特点 |
|------|------|--------|
| N-Gram | 高重复场景（code, structured output） | 高（0.8+），零额外模型 |
| 外置小模型（如 Llama 1B/3B） | 通用对话 | 中（0.5-0.7），需额外 VRAM |
| EAGLE3（Feature-Level Draft） | Code + 通用，高质量 | 高（0.85+），需 fine-tune draft head |
| Medusa（Multiple Draft Heads） | 批量推理 | 中高，多头并行 |
| PARD（Parallel Autoregressive） | 新方法，vLLM 已集成 | 待评估 |

### 系统 Fit 才是真正的难题

这是 2026 这篇文章最重要的洞察：

> "In 2026, the hard part is system fit: draft quality, KV-cache layout, and scheduler behavior matter more than hype."

几个坑：
1. **KV Cache 重复**：naive speculation 会因 draft 序列的 KV 需要额外空间，在 paged attention 中需要特殊处理，否则 cache 膨胀
2. **高 batch 下 speculation 无效**：draft model 消耗的额外 GPU 算力在高并发时反而拖累吞吐，speculation 只在 low-batch 时有效
3. **Benchmark 失真**：SPEED-Bench 证明合成 prompt（短促/固定格式）会夸大 speculation 的收益，生产环境需要 diverse workload 评估

### FlashAttention-3 的工程限制

不是"更快的 attention"通用方案：
- **仅支持 Hopper GPU（H100/H800）**
- CUDA 12.3+ 必须，12.8 推荐
- 依赖 warp specialization + 异步数据搬运 + pipelined matmul-softmax overlap
- 非 Hopper 卡用 FA2 即可，FA3 无法移植

## 三、deepclaude：Agent Loop 的模型替换工程

### 是什么

`aattaran/deepclaude`（1300+ stars，5月3日创建，669 HN pts）：
- 把 Claude Code 的自主 agent loop（工具调用、文件读写、代码执行）接到 DeepSeek V4 Pro 后端
- 支持 OpenRouter 或任何 Anthropic-compatible API
- 宣称 **17x 成本节省**，UX 不变

### 技术意义

这不是简单的 API 替换，而是一种**模型无关的 Agent Loop 架构模式**：

```
Claude Code UX
     ↓
Agent Loop（工具注册、state 管理、retry 逻辑、系统 prompt）
     ↓
[可替换的 LLM 后端]
  - Anthropic Claude（贵，慢）
  - DeepSeek V4 Pro via OpenRouter（便宜，快）
  - 任何 Anthropic-compatible endpoint
```

核心工程问题是：**能力对齐**。Claude Code 的 Agent Loop 依赖：
- 精确的 tool calling 格式
- 特定 system prompt 风格理解
- 遵循严格指令的能力（不自行简化步骤）

DeepSeek V4 Pro 在这些维度已经足够好，所以替换可行。12 个月前这不可能，现在可以了。

### 成本对比（结合之前记录）

| 模型 | Input | Output |
|------|-------|--------|
| Claude Sonnet 4 | $3/M | $15/M |
| DeepSeek V4 Pro | ~$0.14/M | ~$0.28/M |
| 倍率 | **~21x** | **~54x** |

一个典型 Agent session（50K input + 10K output）：
- Claude：$0.15 + $0.15 = $0.30
- DeepSeek：$0.007 + $0.0028 ≈ $0.01
- **节省 ~30x**（更接近真实情况，比宣传的 17x 还高）

### 风险点

1. **任务成功率下降**：复杂推理、模糊指令场景 DeepSeek 出错率更高，需要 retry，抵消部分成本优势
2. **系统 prompt 兼容性**：Claude Code 的 system prompt 针对 Claude 优化，不保证 DeepSeek 完美理解
3. **隐私与合规**：代码和上下文发往 DeepSeek/OpenRouter，企业场景有合规顾虑

## 五、架构反思：对现有项目的映射

### 反思 A：Speculation 现在可以纳入工程实践了

之前在 2026-04-29 的 tech-insights 里把 Speculative Decoding 列为"长期演进"方向。但这篇文章说明：**2026 年 speculation 已经是生产标配，不是 paper 技术**。

调整判断：
- **自托管场景**：vLLM 开箱支持 N-Gram speculation，几乎零成本开启，对 code generation 场景（α > 0.8）立即有效
- **评估前提**：先测量 acceptance rate（α），低于 0.6 就别开，浪费 GPU；高于 0.8 就直接开
- **不适合的场景**：高并发（batch size > 8）、开放对话（α 低）、低延迟需求而非高吞吐需求

**行动项更新：** 自托管时，在 code generation workload 上优先开启 N-Gram speculation，监控 α。

### 反思 B：deepclaude 模式对我们的启发

我们现有的 Agent 系统里，有没有工具调用密集型的后台 pipeline？如果有：
- 分类：交互实时性要求高 vs. 后台批处理
- 实时场景：Claude/GPT-5，用户体验优先
- 后台场景（摘要、分析、标注）：**立刻考虑切换 DeepSeek V4 Pro**

实施路径：
1. 把 LLM 后端抽象成可替换层（不要在业务逻辑里写死 `anthropic.messages.create`）
2. 用 Gateway 层（如 LiteLLM）统一 API，按任务路由
3. 先在低风险后台任务上切换 DeepSeek，观察成功率
4. 建立 per-task 成功率监控，用数据说话

### 反思 C：FlashAttention-3 与我们的关系

现阶段：**无直接关系**（API 调用，硬件由上游决定）。  
但要记住：如果未来自托管，选 H100 级别的卡，FA3 是理所当然要开的。用 A100 的话，FA2 已经够好，FA3 不适用。
