## 二、技术深度笔记

### 2.1 推理成本量化建模——写代码估成本

之前记了理论公式，今天把它整理成可直接用的估算模型：

```python
def estimate_inference_cost(
    model_size_b: float,        # 模型参数量（B）
    tokens_per_request: int,    # 每次请求 token 数
    requests_per_day: int,      # 日请求量
    gpu_cost_per_hour: float = 3.0  # A100 小时成本（USD）
):
    # 经验公式：7B 约 100 tok/s，每 10x 参数约降低 0.6x 效率
    throughput_tps = 100 / (model_size_b / 7) ** 0.6

    total_tokens_per_day = tokens_per_request * requests_per_day
    gpu_hours = (total_tokens_per_day / throughput_tps) / 3600
    daily_cost = gpu_hours * gpu_cost_per_hour
    cost_per_1k = daily_cost / (total_tokens_per_day / 1000)
    return daily_cost, cost_per_1k
```

这个模型可以回答一个实际问题：**什么时候自托管比 API 便宜？**

基准对比（月 token 消耗 100M）：
- Claude Sonnet 4 API：~$300 input + 不定 output
- 自托管 Llama-70B（A100 x4）：GPU 成本约 $1,300/月

结论：月消耗 < 3-5 亿 token 时，API 仍然更便宜（省运维，省工程）。**自托管触发阈值大概在 3-5 亿 tokens/月**。

### 2.2 KV Cache 5 大优化方向（学术整理）

arXiv:2603.20397 把所有 KV Cache 优化技术归纳为 5 类：

| 方向 | 代表方案 | 原理 | 记忆要点 |
|------|----------|------|----------|
| **Cache Eviction** | H2O, StreamingLLM | 驱逐"不重要"的 KV 对 | 适合长文档摘要，代码生成禁用 |
| **Cache Compression** | TurboQuant, KIVI, KVQuant | 量化压缩 K/V 张量 | 无精度损失，模型无关，最安全 |
| **Hybrid Memory** | Infini-Attention, MemGPT | 冷热分层，远程 KV 放 CPU/SSD | 超长 context（1M+ token）专用 |
| **Novel Attention** | MLA (DeepSeek), GQA, MQA | 架构层面减少 KV 头数/维度 | 训练时决定，推理时受益，无法事后加 |
| **Combination** | 压缩+eviction+prefix cache | 多技术叠加 | 最优但实施复杂，需实验验证 |

**论文核心结论：** 没有单一技术在所有场景最优。最优策略 = workload-specific + 多技术组合。这是显而易见但重要的工程告诫：**不要追新技术，追场景适配度**。

### 2.3 七个部署场景的 KV 策略映射（今天最新的框架）

Paper 给出了 7 个典型部署场景及其推荐策略：

```
场景 1: 长上下文单请求 (>1M tokens)
  → 首选 Hybrid Memory（CPU offload + Infini-Attention）
  → Compression 作为补充
  → 不推荐 Eviction（单请求全程需要 context 完整性）

场景 2: 最小化模型修改（想用开箱即用方案）
  → Cache Compression（model-agnostic，无需微调）
  → TurboQuant / KVQuant 系列

场景 3: 高吞吐服务（数据中心推理）
  → Prefix Caching + Continuous Batching 优先
  → 配合 Compression 节省显存 → 塞更多 batch
  → Eviction 按场景叠加

场景 4: 边缘设备 / 内存受限
  → 激进量化（INT4 KV）
  → Novel Attention（GQA/MQA 架构，选择内置这些的模型）
  → 不适合复杂 Combination（计算开销大）

场景 5: 多轮对话
  → Prefix Caching（system prompt 复用）
  → 对话历史选择性 Eviction（早期轮次）
  → 注意 Eviction 可能破坏多轮引用

场景 6: Prefill 密集型（RAG，长输入短输出）
  → Compression 优先（KV 生成量大）
  → 配合 FlashAttention 加速 prefill
  → 如果用 vLLM：开启 prefix caching，固定 system prompt + retrieval 前缀

场景 7: 精度敏感推理（逻辑/法律/医疗）
  → 保守策略：compression only，no eviction
  → 量化精度要高（3.5-bit+ 而非激进 INT2）
  → 禁用 H2O 类 eviction
```

这个框架比我之前自己总结的决策树更系统。我把它合并进 tech-insights 里。

### 2.4 从物理瓶颈看优化方向的推导逻辑

今天有一个思维上的串联：

```
Decode 是 memory-bandwidth-bound
  → 降低 KV 大小 = 降低 HBM 读取量 = 直接提升 decode 速度
  → Compression 的收益不只是"省内存"，同时是"加速 decode"
  → 这解释了为什么 TurboQuant 6x 内存节省 → 不只是 6x 内存，还有吞吐提升

反过来：
  Prefill 是 compute-bound
  → FlashAttention 减少 SRAM/HBM 数据移动，直接降低 IO 开销
  → Prefill 的瓶颈是 IO，不是算力本身
```

理解了物理瓶颈 = 知道在哪儿花刀。

## 三、反思：现有项目架构的问题

### 问题 A：我们缺一个 Workload Profile

目前没有对业务 LLM workload 做分类统计。不知道：
- 平均 input/output token 分布是什么
- 哪些请求是 prefill 密集（RAG）、哪些是 decode 密集（对话生成）
- 多轮对话占比多少，单次调用占比多少

**如果没有 workload profile，所有优化都是猜测。**

按今天 7-scenario 框架，我们最可能的场景是：
- 用户实时对话 → 场景 5（多轮）
- RAG 增强问答 → 场景 6（prefill 密集）
- 后台批处理 → 场景 3（高吞吐）

但比例不清楚 → 对应优化策略不清楚。

**Action：统计一周真实 request 的 token 分布，建立 workload profile。**

### 问题 B：Compression 是当前 API 侧最可行的零风险优化

当前用 Claude/GPT API，不能控制推理引擎。但 prompt 结构上的优化等价于"应用层 compression"：
- context 裁剪 = 减少不必要 KV 生成
- tool result 摘要 = 减少 decode 阶段需要处理的历史 KV
- prompt caching = 最高效的"static KV reuse"

上周已经记了这些，但今天的框架帮助更清楚地定位：**我们做的 prompt 结构优化 = 场景 6 的应用层版本**。

### 问题 C：Combination Strategy 的工程代价被低估

Paper 指出 Combination Methods（多技术叠加）虽然效果最好，但：
- 各技术有兼容性约束（eviction + compression 的参数需联合调整）
- 需要实验验证，不同模型的最优组合不同
- 监控复杂度倍增

**对我们的意义：** 不要急着上复杂组合策略。单技术逐步引入，每次有 baseline 对比，才能知道每个技术的真实收益。
