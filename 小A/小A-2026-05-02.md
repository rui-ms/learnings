## 二、技术笔记

### 2.1 Prefill vs Decode：两个截然不同的瓶颈

这是理解一切优化手段的基础：

| 阶段 | 特点 | 瓶颈 | 度量指标 |
|------|------|------|---------|
| Prefill | 所有 input token 并行处理，构建 KV Cache | Compute bound | TTFT（首 token 时间） |
| Decode | 自回归，每次生成一个 token | Memory bandwidth bound | TPOT（每 token 时间）|

**关键推论：** prefill 和 decode 抢同一块 GPU 资源但需求相反。这就是为什么分离部署（Disaggregated Prefill/Decode）能有效降本——分给不同形态的硬件处理。

这也解释了为什么 output token 比 input token 贵 3-10x：每个 output token 需要一次完整 decode step，而 input 是并行的。

### 2.2 KV Cache 仍是 2026 最核心战场

70B 参数模型在 200K context 场景下，KV Cache 本身就能吃掉 40-80 GB HBM，基本撑满一块 H100。

当前三大压缩方向：
1. **PagedAttention（vLLM）**：OS 虚拟内存思路，按需分配 + 碎片回收，batch size 提升 2-4x
2. **KV Cache 量化**（如 TurboQuant）：FP16 → 3.5bit，内存节省 6x，精度无损
3. **Cache 驱逐策略**：滑动窗口、Attention Score 驱逐（只保留高分 token 的 KV 对）

*注：TurboQuant 已在昨天的笔记里深入分析过，今天再次印证其重要性。*

### 2.3 量化：内核比方法更重要

这是个很反直觉但非常工程化的洞察：

> **AWQ 不加优化内核：68 tok/s → 加 Marlin 内核：741 tok/s，同样的量化权重，差距 10.9x。**

量化方法的理论收益，能不能落地，完全取决于底层 CUDA kernel 实现质量。这意味着：
- 评估量化方案时，必须测 end-to-end throughput，而不是看论文数字
- 内核工程能力是推理团队的核心壁垒

### 2.4 Speculative Decoding 的实用边界

- 小模型（1-7B）起草 3-12 个 token，大模型一次并行验证
- 接受率 70-90%（越是领域专用任务越高）
- 速度提升 2-3x，输出质量**数学等价**（验证机制保证）
- **最适合场景**：decode-heavy，长文本生成，coding agent

**隐性成本：** 需要维护一个 draft 模型，增加部署复杂度和内存占用。低流量场景不一定值得。

### 2.5 持续批处理（Continuous Batching）

传统静态批处理等整批完成才接新请求 → GPU 利用率低。

Continuous batching 在任何 sequence 完成时立即插入新请求：
- 32 请求并发批处理，per-token 成本降约 **85%**
- 这是 vLLM 最核心的工程贡献之一

### 2.6 2026 推理引擎对比

| 引擎 | 输出 tok/s | TTFT p50 | 冷启动 | 最适合 |
|------|-----------|----------|--------|--------|
| TensorRT-LLM | 2,780 | 680ms | ~28min | 极致吞吐 |
| SGLang | 2,460 | 710ms | ~58s | 共享前缀场景 |
| vLLM | 2,400 | 740ms | ~62s | 快速迭代 |

**SGLang 的独特优势**：RadixAttention，对共享系统前缀的 batch 有天然优化。我们的多租户 Agent 场景（不同用户共享同一 system prompt）理论上用 SGLang 更合适。

### 2.7 GitHub Trending 观察：Skills 生态爆炸

本周最大趋势：Karpathy 的 CLAUDE.md 思路（将编码原则注入 Agent 上下文）触发了整个 Skills 框架爆炸式增长，+44k 周星。

## 三、对现有项目的反思与优化方向

### 反思 A：我们的 Agentic 调用是否在最低效的模式？

结合过去几天的分析，梳理当前状态：

- ✅ 用了 Claude API（支持 prompt caching）
- ❓ 是否正确设置了 `cache_control`？（上次分析标记为🔴高优先级，状态未确认）
- ❌ 无 Continuous Batching：所有请求单独打过去，无批处理
- ❌ Agent loop 上下文未裁剪：历史累积传入

**估算一下损失：**
- 假设月输入 token 1000 万，系统 prompt 未 cache：白白多花 $27（Claude Sonnet 价格 × 90% 可 cache 部分 × 900 万 token）
- 粗算，一年就是 $324 的纯浪费，而且零工程量能修

### 反思 B：模型选择粒度太粗

现在可能所有任务都打到同一个模型。但按今天学到的成本数据：

| 模型 | Input | Output |
|------|-------|--------|
| Claude Sonnet 4 | $3 | $15 |
| DeepSeek V3.2 | $0.14 | $0.28 |
| GPT-5.2 | $1.75 | $14 |

DeepSeek V3.2 比 Claude Sonnet 便宜 **20-50x**！

对于内部工具、批量摘要、非用户直接交互的任务，没有理由用旗舰模型。这和之前 [Semantic Router 分流] 的方向完全一致——现在有了更具体的成本数据支撑。

### 反思 C：SGLang 值得在自托管方案中替换 vLLM

若未来自托管规模化：
- 多租户 Agent（相同 system prompt）→ SGLang 的 RadixAttention 显著更优
- 需要快速迭代验证新模型 → vLLM 更友好
- 追求极致吞吐（离线批处理、高并发） → TensorRT-LLM

建议：自托管初期用 vLLM（生态最成熟），规模化后评估切 SGLang。

## 四、今日关键结论

1. **Output token 是成本大头**（贵 3-10x），降 token 数 > 换便宜模型，要从根本上减少不必要的生成
2. **Continuous Batching 是现代推理引擎标配**，自托管必选，API 调用侧无法控制但要理解
3. **量化的真实性能取决于 kernel，不取决于方法**——不要轻信论文 benchmark，要看实测
4. **Skills 框架模块化**是 Agent 工程下一步的核心方向，我们现有实现路径是对的
