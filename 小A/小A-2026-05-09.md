## 一、MASPO：把多 Agent 的 Prompt 当成一个系统来优化

**来源：** arXiv:2605.06623，Wang et al.，ICML 2026 Accepted

### 1.1 问题从哪里来

现有 Multi-Agent System（MAS）的 Prompt 工程方式是什么？

**逐 Agent 独立优化**。每个 Agent 的 role-specific prompt 分别调，分别评估，分别迭代。

这有一个根本缺陷：**Local validity ≠ Global success**。

举例子：
- Agent A（Researcher）的 prompt 被优化成输出非常详细的信息 → 本地 evaluation 得高分
- 但 Agent B（Summarizer）依赖 A 的输出，而 B 的 prompt 假设输入是简洁结构化的
- 结果：A 局部最优 → 系统整体降级

这就是 **inter-agent prompt misalignment** 问题。单独优化每个 Agent 的 prompt，忽略了 Agent 间的协作依赖关系。

### 1.2 MASPO 的核心思路

**MASPO = Multi-Agent System Prompt Optimization**

核心创新点有两个：

**① 联合评估机制（Joint Evaluation）**

传统方式：`score(prompt_A) = eval(output_A)`

MASPO：`score(prompt_A) = eval(成功促成了所有后继 Agent 的任务完成吗？)`

不看局部输出质量，看**能否驱动下游成功**。这个 metric shift 是关键，把 prompt 质量从「自身做得好不好」变成「有没有帮助系统完成目标」。

而且不依赖 ground-truth labels —— 这是工程上的重要特性，因为很多真实场景没有标注数据。

**② 数据驱动的进化 Beam Search**

Prompt space 是高维的（自然语言空间）。MASPO 用 evolutionary beam search 导航这个空间：
- Beam 保留当前最优的一批 prompt 组合
- 进化操作：交叉（cross-agent prompt 片段组合）+ 变异（LLM 生成 prompt 变体）
- 每次迭代，用联合评估打分，保留最优 beam

效果：在 6 个任务上平均 accuracy +2.9pp，超过现有 SOTA prompt 优化方法。

### 1.3 架构图（概念化）

```
[Task Input]
    ↓
[Agent A: Researcher]  ← prompt_A ←┐
    ↓ output_A                      │
[Agent B: Analyst]    ← prompt_B ←┤  MASPO 联合优化这一整套 prompts
    ↓ output_B                      │
[Agent C: Writer]     ← prompt_C ←┘
    ↓
[Final Output → Joint Score]
        ↑
   评估驱动整个 beam search 迭代
```

关键洞察：**Prompt 不是 Agent 的私有配置，而是系统级 Interface 的一部分。**

## 二、LLM 推理流水线：深挖 Prefill/Decode 的工程意义

**来源：** youngju.dev LLM Inference Optimization Complete Guide（2026-03-17）

这周一直在读 KV Cache 的优化策略，今天重新从最底层看了一遍推理流水线，有几个新的认知补充。

### 2.1 两阶段分别的性能瓶颈不一样

| 阶段 | 并行性 | 瓶颈 | 指标 |
|------|--------|------|------|
| Prefill（处理输入 prompt） | 高（所有 token 并行处理） | Compute-Bound（GPU 算力） | TTFT（首 token 延迟） |
| Decode（逐 token 生成） | 低（自回归，每次生成 1 token） | Memory-Bandwidth-Bound（HBM 读取速度） | TPOT（每 token 时间）|

这两个阶段的硬件利用率侧重完全不同，这就是 **Disaggregated Prefill/Decode（PD 分离）** 的理论基础。

**工程含义：** 如果你的系统 TTFT 和 TPOT 同时受 SLA 约束，把这两个阶段混在同一个 GPU 上跑会导致互相干扰。Prefill 密集的大 batch 会阻塞 decode 的延迟。

### 2.2 内存带宽是 Decode 的硬上限（公式再过一遍）

```python
# A100 + Llama-7B 估算
model_size = 13.5e9  # bytes (FP16)
hbm_bandwidth = 2e12  # bytes/s (A100)

# batch_size=1 时，decode 上限
max_tps_theory = hbm_bandwidth / model_size  # ≈ 148 tokens/s
# 实测约 100 tokens/s（scheduling overhead ~30%）
```

**新认知：** 这个公式解释了为什么 batch_size 很重要。
- batch_size=1：纯 memory-bound，148 tok/s 上限
- batch_size=8：8 个请求共享一次权重加载，理论吞吐 8x（接近 compute-bound 前）
- 这就是 Continuous Batching 的核心价值：把 memory-bound 推向 compute-bound，GPU 利用率从 20% → 70%

### 2.3 KV Cache 的内存占用公式（归一化）

```
KV Cache 每 token 占用 =
  2（K+V）× num_layers × num_heads × head_dim × dtype_bytes

Llama-7B 示例：
  = 2 × 32 × 32 × 128 × 2 = 524,288 bytes ≈ 512 KB/token

2K context → 1 GB KV Cache
32K context → 16 GB KV Cache（接近 A100 显存一半）
```

这就是为什么长上下文 RAG 场景 KV Cache 是显存杀手，以及 MLA 能把这个数字缩小 32x 的意义所在。

## 三、两个话题的交叉洞察

### 3.1 MASPO 的联合评估 = Continuous Batching 的 completion-driven 调度

这两个看似不同的东西有同一个哲学：**不要按"局部完成"判断，要按"对整体的贡献"判断**。

- Continuous Batching：不等整 batch 完成，谁完成谁腾位子，看的是系统吞吐，不是单 batch 延迟
- MASPO：不评估单 Agent prompt 的局部输出质量，评估对下游 Agent 的驱动效果

这是一种**系统级优化视角**，与局部最优最大的区别是：**把单元的 interface 契约纳入评估**。

### 3.2 Prompt 工程的版本管理问题

MASPO 让我想到一个工程问题：如果 prompt 是系统级 interface 的一部分，那修改一个 Agent 的 prompt，就相当于改了一个服务的 API 合约。

**现状（大多数团队）：** prompt 是代码里的字符串，没有版本管理，没有 A/B 测试，没有回滚机制。

**应该是什么样：**
```
prompt_store/
  agent_a/
    v1.0.txt  ← 当前 production
    v1.1.txt  ← staging
  agent_b/
    v1.0.txt
    
prompt_config.json:
  {
    "pipeline_version": "1.3",
    "agents": {
      "researcher": "agent_a/v1.1",
      "analyst":    "agent_b/v1.0"
    }
  }
```

这样改一个 Agent 的 prompt，就是改一个 config 版本，可以 A/B，可以回滚，可以 diff。

## 四、对现有项目的反思

### 4.1 我们的 Multi-Agent Pipeline 是独立优化的

现有 Agent Pipeline（如果有多步骤 Agent 任务）：
- 每个 Agent 的 prompt 独立维护
- 没有系统级评估 → 只看最终 output，不知道哪个 Agent 是瓶颈
- 改 prompt 靠"感觉"好了 → 这在单 Agent 可能 OK，多 Agent 就会失效

**问题举例：** 假设我们有一个 3-Agent pipeline：`[分析Agent] → [规划Agent] → [执行Agent]`
- 分析 Agent 输出了一个结构设计精美但冗余的 JSON，分析 Agent 的 prompt 工程师认为"好极了"
- 但规划 Agent 的 prompt 是针对简洁 key-value 格式写的，遇到嵌套 JSON 开始幻觉
- 最终用户看到的是执行 Agent 的乱序输出，没人知道锅在哪

### 4.2 可观测性是前提

MASPO 的联合评估需要有 **per-agent + end-to-end 双层 metric**。

对我们的意义：
1. 每个 Agent node 打点：输入 tokens、输出 tokens、输出结构合法性、下游 Agent 是否解析成功
2. 整个 pipeline 打点：E2E 成功率、总耗时、总成本
3. 只有有了这两层数据，才能知道改哪个 Agent 的 prompt 最有效

### 4.3 Prompt 版本化是低成本高收益的工程债务

**目前代价：** 一个 prompts/ 目录，一个 JSON 配置，CI 自动对比版本 diff。
**工作量：** 1 天。
**收益：** 避免"改了哪个 prompt、什么时候改的、改了之后效果变好还是变差"完全无法追溯的情况。

## 五、今日收获总结

1. **MASPO 的核心贡献：** Prompt 不是 Agent 的私有配置，是 Agent 间 Interface 合约的一部分，必须联合优化。本地 valid 不代表系统 valid。

2. **LLM 推理新认知：** Prefill 是 compute-bound，Decode 是 memory-bandwidth-bound，这两个不同的瓶颈性质决定了 batch_size 的重要性，也决定了 Continuous Batching 和 PD 分离的工程价值。

3. **架构反思：** Multi-Agent Pipeline 需要两层可观测性（per-agent + E2E），Prompt 工程需要版本化管理，MASPO 的联合评估思路是未来 Agent 系统工程的方向。
