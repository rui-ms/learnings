## 今天学了什么

### 一、KV Cache 的五个优化方向（arxiv 2603.20397）

这篇 paper 是目前最系统的 KV Cache 优化综述，24 页，14 张图，把所有现有方案整理进了 5 个方向：

#### 1. Cache Eviction（缓存驱逐）
当上下文超长、HBM 放不下时，选择"驱逐"哪些 token 的 KV。核心假设：不是所有 token 的 attention score 都重要。
- **H2O（Heavy Hitter Oracle）**：统计哪些 token 被高频 attend 到，保留 heavy hitter，驱逐 rest。实验上 80-95% 的 KV 可安全驱逐。
- **StreamingLLM**：只保留 attention sink（通常是前几个 token）+ 滑动窗口，实现无限长度 streaming，质量损失可接受。
- **局限性**：不可逆——被驱逐的 token 重新出现在对话里会有幻觉风险。适合单次长文档 summarization，不适合多轮对话。

#### 2. Cache Compression（缓存压缩）
保留所有 token，但压缩每个 token 的 KV 表示：
- **量化（KV Quantization）**：FP16 → INT8/INT4，内存减半或更多，结合旋转操作（如 TurboQuant 的随机正交旋转）可做到无精度损失。
- **Low-rank Approximation**：将 KV 矩阵分解为低秩形式，减少存储维度。

#### 3. Hybrid Memory（混合存储）
KV Cache 不局限于 GPU HBM，分层存储：
- GPU HBM（热 cache）→ CPU DRAM（温 cache）→ NVMe SSD（冷 cache）
- 挑战：GPU ↔ CPU 的 PCIe 带宽（约 64 GB/s）远低于 HBM（3.35 TB/s on H100），频繁换页有严重延迟
- 实用场景：超长单文档推理（>1M token），单对话并发极低时可用

#### 4. Novel Attention Mechanisms（从架构层规避问题）
改 attention 本身：
- **GQA（Grouped Query Attention）**：多个 query head 共享同一对 KV head，内存线性缩减。Llama 3 系列默认采用。
- **MLA（Multi-head Latent Attention）**：DeepSeek V2/V3 创新，将 KV 压缩成低维 latent 向量存储，推理时解压，KV Cache 内存减少 5-13x，吞吐 5x 提升。
- **MLA 的意义**：这是架构层的根本性改变，不是 post-hoc 优化补丁。

#### 5. Combination Strategies
没有 silver bullet，最优解是组合：
- Prefill 阶段：Flash Attention（IO-aware compute）+ GQA（减少 KV head）
- Decode 阶段：PagedAttention（内存碎片）+ 量化（HBM 节省）+ Cache Eviction（超长 context）
- Paper 的结论：**没有单一技术在所有场景都最优，workload-adaptive 多阶段 pipeline 是未来方向**

### 二、Continuous Batching 的生产工程细节

Morph Guide 里的数据值得细抠：

**传统 Static Batching 的问题：**
- 一批请求里最长的序列决定 padding 长度 → GPU 利用率低
- 最短的请求完成后必须等最长的 → 队头阻塞（Head-of-Line Blocking）
- 大 batch 下内存碎片严重

**Continuous Batching（iteration-level batching）解决了什么：**
- 每个 decode step 后，已完成的序列立即腾出位置，新请求插入
- GPU 利用率近 100%（vs 静态 batching 的 30-60%）
- 结合 PagedAttention：无需预分配最大 context 长度的连续内存

**量化的关键认知：同样的 weights，kernel 决定一切**

| 方法 | 无优化 kernel | Marlin kernel | 倍率 |
|------|-------------|---------------|------|
| AWQ 4-bit | 68 tok/s | 741 tok/s | 10.9x |

同一份量化权重，有没有优化 kernel，差 10x 吞吐。这意味着选推理引擎时，不只看量化支持，更要看 kernel 优化水平。vLLM 和 SGLang 的 Marlin kernel 支持是一线特性，自托管时必须开启。

### 三、Disaggregated Prefill/Decode：物理分离的意义

这是 2026 年正在生产化的方向：

**为什么要分离？**
- Prefill：计算密集（处理所有 input token 并行）→ 需要高 FLOPS
- Decode：内存带宽密集（每 step 读一遍全量 KV Cache + weights）→ 需要高 HBM 带宽

当这两个 phase 混在同一张卡上，互相抢占资源，都不在最优工作点。

**分离方案：**
- **Prefill 节点**：H100 SXM（高 FP16/FP8 FLOPS）
- **Decode 节点**：H200（HBM3e，内存更大带宽更高）或多张 A100

**实际收益（Morph 数据）：**
- 混合部署：TTFT 受 decode 干扰，TPOT 受 prefill 干扰
- 分离后：TTFT 降 30-40%，TPOT 改善 20-30%，总 GPU 成本可降约 20%

**门槛：** 需要高速互联（NVLink 或 InfiniBand），工程复杂度显著上升，适合月 token 消耗 > 5亿的规模。

## 对现有项目的反思

### 目前的状态（截至今天的 tech-insights 积累）

- ✅ 已识别：模型分层调用的必要性（DeepSeek vs Claude）
- ✅ 已识别：Prompt Caching 的 ROI 极高（输入成本降 90%）
- ✅ 已识别：Agentic Loop context 精简
- ❓ 未评估：KV Cache 是否在我们自己有控制权的地方？

### 新的思考

**MLA 架构的选型影响**

DeepSeek V3.2 已是 $0.14/$0.28 的价格梯队，且其 MLA 架构使自托管时的 KV Cache 内存消耗远低于等 size 的 Llama/Mistral 架构模型。如果未来要自托管，**DeepSeek 架构的模型在推理成本上有结构性优势**，不只是价格，而是硬件效率。

**Continuous Batching 与我们的 Agent 架构的关系**

当前 Agent pipeline 是单请求串行的（每步 LLM call 独立发出）。如果迁移到自托管的 vLLM/SGLang，我们的 Agent step 之间有天然的等待窗口，这些窗口可以被 continuous batching 填满——意味着 GPU 利用率实际上比 naive 估算高，自托管的 break-even 点比想象中低。

**Combination Strategy 的工程启示**

Paper 的结论"没有单一技术适配所有场景"对应到工程层面是：**推理优化不是一次性配置，是 workload-aware 的持续调优**。
- 短对话 → 简单 batching + 量化即可
- 长文档 RAG（>64K token）→ 必须加 cache eviction
- 多轮代码生成（超长 context window）→ MLA 架构模型 + hybrid memory

我们目前的业务场景分析还停留在"有 RAG"的粒度，需要进一步细分 workload profile，才能对号入座地选择优化组合。

## 今天最有价值的 3 个认知

1. **MLA 是架构层的革命，不是优化补丁。** DeepSeek 选择从 attention 机制本身解决 KV 问题，而不是 post-hoc 压缩，这是设计哲学的差别。未来选模型时，训练效率之外，KV 效率是第二个维度。

2. **Kernel 比量化方法更重要。** AWQ 4-bit 无 Marlin kernel 是 68 tok/s，有 Marlin kernel 是 741 tok/s。选推理栈时，别只看量化 bit，看 kernel 是否跟上。

3. **Disaggregated Prefill/Decode 正在生产化，但门槛仍高。** 2026 年这个方向已有 Mooncake（月之暗面）等公司在大规模落地，但需要 NVLink/InfiniBand 支持。我们的介入点：关注 vLLM 的 PD 分离 API（预计 Q3 2026），提前了解接口，规模化时可快速切入。
