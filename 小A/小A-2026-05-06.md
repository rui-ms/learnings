## 二、Prefill vs Decode：两个截然不同的计算性质

LLM 推理是两个阶段拼在一起的过程，但它们的瓶颈完全不同：

| 阶段 | 工作内容 | 瓶颈类型 | 核心指标 |
|------|----------|----------|----------|
| Prefill | 并行处理 prompt 所有 token | **Compute-Bound**（矩阵乘法） | TTFT（首 token 延迟） |
| Decode | 自回归，一次生成一个 token | **Memory-Bandwidth-Bound** | TPOT（每 token 时间） |

**Decode 为什么是内存瓶颈？** 用一道简单的数学来看：

- Llama-2 7B：32层 × (4096² × 4 + 4096² × 8) × 2 bytes ≈ **13.5 GB** 权重
- A100 HBM 带宽：2 TB/s
- batch_size=1 时，每个 token 都要把全部 13.5 GB 权重从 HBM 读一遍
- 理论上限：**2TB/s ÷ 13.5GB ≈ 148 tokens/s**
- 实测 A100 约 100 tokens/s — 和理论基本吻合

**关键洞察：** Decode 阶段基本上是在做"读内存"，GPU 算力 99% 都在等数据。
这就是为什么 KV Cache 压缩（减少 HBM 读取量）和 batching（分摊权重读取）是两条正确路：一条降分子，一条提分母。

## 三、Continuous Batching：GPU 利用率的真正来源

### 3.1 Static Batching 的根本问题

最早的推理服务：把一批请求打包成 batch，等所有请求都完成才出结果。

问题在哪？每个请求生成的 token 数量不同：
```
请求 A：生成了 10 个 token（完成了）
请求 B：还在生成...（还需要 90 个 token）
请求 C：还在生成...
```
A 完成后，GPU 里这个 slot 一直空着等 B、C，**实际利用率可能只有 20-30%**。

### 3.2 Continuous Batching 的核心逻辑

解法很优雅：**每生成一个 token 后，检查是否有完成的请求，立刻用新请求填进来。**

不再等整个 batch 完成，而是 token 粒度的动态调度。
- 吞吐量提升：5-10x（来自 vLLM 论文数据）
- GPU 利用率：从 20-30% 提升到 70-80%

```
时间轴：
t=0: [A, B, C, D]
t=1: A 完成 → [E, B, C, D]  ← 立刻插入新请求
t=2: B 完成 → [E, F, C, D]
```

**这才是 vLLM 的第一个核心创新。** 不是 PagedAttention，是 continuous batching + iteration-level scheduling。

### 3.3 Iteration-Level vs Request-Level 调度的工程含义

从系统架构角度，iteration-level 调度意味着：
- 调度器必须在毫秒级别做 preemption 决策
- 需要知道每个请求当前的 KV Cache 状态
- 需要 memory 层面的精细化管控（否则 OOM 了就完了）

这就自然引出了下一个问题：**怎么管内存？**

## 四、PagedAttention：把操作系统的分页内存带进推理引擎

### 4.1 传统 KV Cache 的内存碎片问题

每个请求需要连续的 KV Cache 内存。但请求的生成长度不可预知，所以必须预先分配最大可能长度的内存（max_seq_len × layers × heads × head_dim × 2 × 2 bytes）。

对 Llama-2 7B，max_seq_len=4096：
```
KV Cache/请求 = 2 × 32层 × 32头 × 128维 × 2bytes × 4096 tokens = 2 GB
```

结果：
- 90% 的请求用不到 2GB（大多数对话很短）
- 但必须预分配 → 严重的内存碎片 → 实际并发请求数远低于理论值

### 4.2 PagedAttention 的设计

借鉴操作系统虚拟内存：**把 KV Cache 切成固定大小的 block（16 tokens），按需分配，不连续也没关系。**

- Block table 记录每个请求的逻辑 block → 物理 block 映射
- 新请求来了分配 block，生成结束立刻回收
- 内存浪费从 ~60% 降到 < 4%

**隐含的好处：Copy-on-Write 语义支持 beam search 和多个并发请求的 KV 共享（prefix caching 的底层基础）。**

这就是为什么之前学的 vLLM prefix caching 要求 16-token block 对齐——它直接建在 PagedAttention block 的 hash 之上。

```
逻辑视角：
请求A: [block_0, block_1, block_2, ...]

物理视角（PagedAttention）：
block_0 → GPU 内存地址 #47
block_1 → GPU 内存地址 #193
block_2 → GPU 内存地址 #12
（不连续，但逻辑上是连续的）
```

## 五、Multi-GPU 推理：Tensor Parallel vs Pipeline Parallel

这是今天新啃的一块内容。单 GPU 搞不定大模型时，怎么切分？

### 5.1 Tensor Parallelism（TP）

**核心思路：** 把单层的权重矩阵在多个 GPU 上切分，每个 GPU 算部分结果，最后 AllReduce 合并。

```
GPU 0: W_Q[:, :2048]   GPU 1: W_Q[:, 2048:]
每个 GPU 算部分 attention heads
→ AllReduce → 完整输出
```

**特点：**
- 延迟低（层内同步，完成度高）
- 通信量大（每层都有 AllReduce，需要高带宽互联）
- NVLink 是 TP 的命脉，PCIe 跑 TP 性能崩

**适合：** 单个请求的延迟优化（对话场景，TTFT 敏感）

### 5.2 Pipeline Parallelism（PP）

**核心思路：** 不同 GPU 负责不同层（前 16 层在 GPU 0，后 16 层在 GPU 1）。

```
GPU 0: layers 0-15 → 传激活值 → GPU 1: layers 16-31
```

**特点：**
- 通信量小（只传激活值，一次性）
- 延迟高（micro-batch 流水线，但不能完全消除 bubble）
- 对互联带宽要求低，PCIe 也可以跑

**适合：** 多请求并发吞吐优化（推理工厂场景，GPU 多）

### 5.3 实践选型

| 场景 | 推荐策略 | 原因 |
|------|----------|------|
| 2-4 GPU，对话实时服务 | TP | 低延迟，NVLink 利用率高 |
| 8+ GPU，高并发推理 | TP + PP（混合） | 先 TP 在节点内，PP 跨节点 |
| 无 NVLink（云 V100） | PP 或量化单卡 | PCIe 跑 TP 得不偿失 |
| 权重太大单卡放不下 | TP（必须） | 没得选，但要评估通信成本 |

```
推理优化全景
│
├── 1. 调度层（GPU 利用率）

│   └── PagedAttention → 碎片 <4%，prefix 共享

│

│
└── 5. 集群架构层（多卡多机）
    ├── TP → 低延迟，需 NVLink
```

## 七、对现有项目的反思

### 当前状态

项目目前是 API 调用模式，还没有自托管推理。所以上面大量内容（TP/PP、kernel 选型）现在用不到。

### 但有几件事现在就能做：

**1. Continuous Batching 思路应用于业务层**

我们在应用层其实也有类似问题：多个并发的 Agent 任务，彼此独立，各自调用 LLM。
目前的架构是"请求串行发出"——如果能在业务层实现类似 continuous batching 的思路（任务完成就提交新任务，而不是等整批），整体吞吐会提升。

**→ Action：** 梳理现有 Agent pipeline，识别是否有任务等待/空转现象。

**2. PagedAttention 的思路：内存利用率 = 有效并发数**

API 调用模式下，我们也有一个类似的问题：`max_tokens` 设置。
很多代码里 max_tokens 设得很大（比如 4096），但实际输出很短（200 tokens）。
API 侧虽然按实际 token 计费，但对于自托管将来的设计要记住：**max_tokens 是内存预分配的上限，不是实际消耗。**

**3. Memory Bandwidth 分析工具化**

今天看到这段代码挺有价值——可以直接拿来评估不同场景下的推理成本：
```python
memory_bound_tps = bandwidth_tbs * 1e12 / model_weight_bytes
compute_bound_tps = compute_flops / (2 * model_weight_bytes)
bottleneck = "memory" if memory_bound_tps < compute_bound_tps else "compute"
```
**这个模型可以用来快速判断：某个模型在某类 GPU 上的理论极限，从而判断当前推理效率是否达标。**
