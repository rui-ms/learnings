## 核心收获

### 1. WRP 三维框架（Workload-Router-Pool）

vLLM 团队在过去一年做了大量子系统研究后，把 LLM 推理优化抽象成一个三维正交坐标系：

| 维度 | 内容 |
|------|------|
| **Workload** | 请求特征：chat vs. agent、单轮 vs. 多轮、prefill-heavy vs. decode-heavy |
| **Router** | 调度策略：静态语义规则、在线 bandit 自适应、RL 模型选择、质量感知级联 |
| **Pool** | 资源池：同构/异构 GPU、disaggregated prefill/decode、KV-cache 拓扑 |

**架构师视角：** 这个三维分解非常干净。现有大部分团队只关注了 Pool（加机器/换卡），忽视了 Router 层的精细化调度和 Workload 的分层治理。在实际项目里，3x3 的 WRP 交叉矩阵中，多数团队只覆盖了对角线上几个格子。

**关键空白：** 论文明确指出 multi-turn context memory + safety routing 在 agentic workload 下仍是 open problem。

---

### 2. Inference 为什么贵？根本原因

- **Memory-Bandwidth Bound，不是 Compute Bound**：70B 模型 FP16 精度下，单次 forward pass 就要读取 140GB 参数，H100 内存带宽 2TB/s → 光加载权重就 70ms。
- **自回归串行性**：token by token，100 token = 100 次 forward pass，无法并行化。
- **GPU 利用率低**：小 batch size 时 H100 实际算力利用率可能只有 20%。

---

### 3. 最重要的单点优化：Continuous Batching + PagedAttention

- **Static Batching 的问题**：慢请求 block 快请求，GPU idle 时间长。
- **Continuous Batching**：请求完成即补位，batch size 恒定，吞吐提升 2-5x。
- **PagedAttention**：把 KV-cache 按 page 管理（类比 OS 内存分页），消除碎片化，支持跨请求 KV-cache 共享。

**生产坑：** batch timeout 过激进（50ms）→ P99 飙到 8s。调整到 500ms 并依据请求分布 tuning 后 P99 稳定在 1.5s 以内。这种"参数敏感性"在上线前很难通过压测完全覆盖，需要线上灰度 + 监控闭环。

---

### 4. 三种推理模式的场景分工

| 模式 | 优化目标 | 典型场景 |
|------|----------|----------|
| Online Inference | 延迟 < 1s | 聊天、代码补全 |
| Batch Inference | 成本/吞吐（延迟 10-30s 可接受） | 文档处理、邮件摘要、内容审核队列 |
| Streaming Inference | TTFT（首 token 延迟） | 对话 AI、写作助手 |

**实践建议：** 生产系统通常三者混用——用户消息走 Streaming，后台 summarize/embedding 走 Batch，这样成本和体验都能兼顾。

## 对现有项目的反思

### 现状诊断

我们目前的 AI 功能（推测是 RAG + 对话类场景）大概率处于以下状态：
- 直接调 OpenAI/Azure OpenAI API，没有独立的推理层
- 所有请求走同一个 endpoint，没有按 workload type 分流
- KV-cache、批处理策略完全交给上游 API，自己没有控制权
- 成本监控粒度粗（按月账单），缺乏 per-request level 的可观测性

### 可以改进的方向

**短期（可立即做）：**
1. **请求分层**：区分 latency-sensitive（用户实时对话）和 throughput-sensitive（后台任务）两条通道，后者走异步 batch，成本直降 30-50%。
2. **Prompt 优化 + Caching**：对高频重复 system prompt 引入语义缓存（vLLM 已支持 category-aware semantic caching），同类请求命中率高时省一大笔。
3. **可观测性补全**：在 LLM 调用层加 Prometheus metrics（TTFT、TPS、token count、cost/request），不然优化无从下手。

**中期（需要架构投入）：**
4. **引入 Router 层**：在应用层和 LLM 服务之间加一个 semantic router，按请求类型路由到不同模型（大模型 vs. 小模型 vs. 本地模型），走 WRP 框架的 Router 维度。
5. **评估自托管可行性**：若月请求量超过 5000 万 tokens，自托管 Llama 70B + vLLM 的 TCO 可能优于 API 消费，值得做专项 ROI 分析。

**长期（架构演进）：**
6. **Disaggregated Prefill/Decode**：prefill 计算密集、decode 内存密集，分离部署可以针对性优化硬件选型，这是 WRP 框架中 Pool 维度的高阶玩法。
7. **Speculative Decoding**：用小模型做 draft、大模型验证，在不损质量的前提下提速 2-3x，适合 decode-heavy 的长文生成场景。
