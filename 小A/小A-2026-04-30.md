## 二、Granite 4.1 深读笔记

### 核心结论

Granite 4.1 的 8B Dense 模型在多个 benchmark 上超越了上代 32B MoE（9B active）模型。结论很刺激：**参数规模不是瓶颈，数据质量和训练配方才是**。

### 工程方法拆解

#### 1. 五阶段渐进式预训练

| 阶段 | 重点 |
|---|---|
| Phase 1 | 宽泛：CommonCrawl 59%，代码 20%，数学 7% |
| Phase 2 | 深化能力：数学 35%，代码 30% |
| Phase 3-4 | 注入 CoT 轨迹 + 高质量指令数据 |
| Phase 5 | 扩展上下文窗口至 512K |

**架构启示**：训练数据的课程安排（Curriculum）比单纯堆数据量更重要。先建立广泛世界模型，再专项强化，符合人类学习认知规律。

#### 2. LLM-as-Judge 数据过滤管线

IBM 在 fine-tuning 之前，用另一个 LLM 对所有 SFT 样本做六维评分：
- Instruction following / Correctness / Completeness
- Conciseness / Naturalness / Calibration（即不要过度自信）

关键细节：**部分维度没有容错** —— 幻觉、前提错误、计算错误，直接扔掉，不看分数。

最终 SFT 数据集：4.1M 精选样本（相比动辄上亿 token 的粗粒度做法，量少质精）。

#### 3. 四轮 RL 的教训

这是最值得记住的部分。训练过程暴露了一个**能力正交性问题**：

- Stage 2 RLHF 做 general chat，AlpacaEval 涨了 18.9 点
- **但数学 benchmark（GSM8K、DeepMind-Math）同时下滑了**

IBM 的处理方法：
1. Stage 3：短暂的 identity/knowledge calibration（40步）—— 稳定模型自我认知
2. Stage 4：专项 Math RL 补救，GSM8K 恢复并超过了 fine-tuned baseline

**架构启示**：多目标 RL 中的能力退化（Capability Regression）是真实问题，不是理论担忧。需要设计检测机制（专项评测集）+ 补救流程。

### 对我们项目的反思

**问题**：我们现在调用的基础模型是否在企业场景做过类似的专项优化？如果没有，直接 prompt engineering 能覆盖多少？

**建议**：
- 在 production inference pipeline 中，考虑引入轻量的**输出质量校验层**（类似 LLM-as-Judge 思路），特别是结构化输出（JSON schema、tool calls）场景，用 rule-based checker 做后置验证
- 如果有 fine-tuning 计划，数据配方要阶段化，不要一次性混在一起训练

## 三、TradingAgents 多智能体架构分析

### 框架定位

TradingAgents 模拟真实交易公司的组织结构，用多个专职 LLM Agent 协作完成投资决策：
- **分析师层**：基本面、情感、新闻、技术指标 → 4 个专职 Agent
- **研究员层**：Bull/Bear 对抗辩论 → 降低单点偏见
- **交易员层**：综合报告 → 决策
- **风控层**：独立评估 → 最终 Portfolio Manager 审批/拒绝

### 架构亮点

#### 1. 职责分离 + 对抗机制

Bull/Bear 两个 Researcher 的设计很有意思：不是让一个 Agent 自己做 pros/cons，而是两个独立 Agent 各持立场辩论。

**为什么更好**：单个 LLM 做 pros/cons 时会出现"先入为主"偏差（生成顺序影响权重）。对抗机制能更充分地暴露风险，类似 Red Team 的思路。

#### 2. LangGraph 状态机 + Checkpoint Resume

v0.2.4 引入了 LangGraph checkpoint resume，意味着：
- Agent 流程可以在任意 node 中断并恢复
- 不需要从头重放整个 pipeline

**架构启示**：长流程多 Agent 系统必须考虑**幂等性**和**中断恢复**。如果我们的 Agent pipeline 有超过 3 个串行步骤，应该引入类似 checkpoint 机制，避免前端步骤已完成、后端崩溃导致全部重来。

#### 3. 结构化输出 Agent

v0.2.4 的 "structured-output agents" 专门提了 Research Manager、Trader、Portfolio Manager 使用结构化输出。

这说明 LLM 的 free-form 输出在多 Agent 链路中是脆弱点 —— 下游 Agent 消费上游输出时，JSON schema 约束是必要的工程保障，不是"可选优化"。

## 四、今日核心洞察

### 洞察 1：参数效率革命正在发生

从 GPT-4 到 Gemini Ultra 时代"大力出奇迹"，到现在 Granite 4.1 这种**数据工程密集型**的路线，AI 模型训练的竞争维度在转移。

对工程侧的影响：小模型（8B级）+ 精细微调，越来越有可能在特定场景打败大通用模型。**部署成本和延迟的方程式正在重写**。

### 洞察 2：Multi-Agent 架构需要"组织设计思维"

TradingAgents 的成功案例告诉我们：设计多 Agent 系统时，不是"怎么拆 prompt"，而是"怎么设计组织"：
- 每个角色的职责边界清晰吗？
- 有没有对抗/校验机制？
- 决策链路有没有独立的审批节点？
- 中间状态能不能持久化和恢复？
