## 二、核心技术深度笔记

### 2.1 Prefill vs Decode：两个完全不同的瓶颈

```
Prefill 阶段
- 并行处理所有输入 token
- 计算密集（Compute-Bound）
- 瓶颈：GPU 算力（TFLOPS）
- 影响指标：TTFT（Time To First Token）

Decode 阶段
- 每次只生成 1 个 token（自回归）
- 内存带宽密集（Memory-Bandwidth-Bound）
- 瓶颈：HBM 读取速度
- 影响指标：TPOT（Time Per Output Token）
```

**关键数字（Llama-2-7B on A100）：**
- 模型权重：~13 GB
- KV Cache per token：2 × 32层 × 32头 × 128维 × 2bytes = **524 KB/token**（！）
- 2048 token context → KV Cache ~1 GB
- A100 内存带宽：2 TB/s → decode throughput 上限约 160 tok/s（batch_size=1）

> 📌 **洞察**：KV Cache 是 LLM 内存压力的真正来源，不是模型权重。长上下文场景里，KV Cache 甚至比模型参数占更多显存。

### 2.2 H2O：Heavy-Hitter Oracle（Cache Eviction）

**核心观察**：Transformer 注意力矩阵是极度稀疏的，幂律分布——少数 token 持续吸引大量注意力，这些就是 "heavy hitters（H2 token）"。

**H2O 策略：**
```
保留 = Recent tokens（最近 k 个）+ Heavy Hitters（高累积注意力分数的 token）
驱逐 = 注意力分数低且不是 recent 的 token
```

**算法本质：** 动态 submodular 优化问题——在有限 cache 预算内最大化"被保留 token 能覆盖的注意力分数总量"。

**实测收益：**
- LLaMA/OPT 上，保留 20% token → 精度损失 < 1%
- KV Cache 内存减少 5x
- 吞吐提升 2-3x（更多请求可并发）

**适用场景限制：**
- ✅ 长文档摘要（>4K token）
- ✅ 文章/故事生成
- ❌ 精确问答（需要全局上下文的场景精度下降明显）
- ❌ 代码生成（变量引用需要精确的远距离 token）

### 2.3 SGLang RadixAttention vs vLLM Block-Level Prefix Caching

这两种方案解决同一个问题：多个请求共享相同前缀时，不重复计算 KV，直接复用缓存。

#### vLLM 方案：Block-Level Hash（简单但有对齐损耗）

```
原理：
- 将 token 序列划分为固定大小的 block（默认 16 token/block）
- 对每个 block 计算 hash（token IDs 的 hash）
- 请求进来时，逐 block 查找已有 cache，匹配则复用

优点：
- 实现简单，内存管理是 PagedAttention 的自然延伸
- cache 查找 O(1)（hash lookup）

缺点：
- block 边界对齐问题：如果前缀不是 16 的倍数，最后一个不完整 block 无法缓存
- 粒度粗，"接近"但不完全一致的前缀无法复用
- 最坏情况：浪费 15 个 token 的对齐损耗
```

#### SGLang 方案：RadixAttention（精细但复杂）

```
原理：
- 用 Radix Tree（前缀树）存储所有已缓存的 token 序列
- 每个树节点 = 一段 token 序列 + 对应的 KV 张量指针
- 新请求进来时，沿树做最长公共前缀匹配
- 匹配长度最大化 → 最小化重新计算量

优点：
- Token 级粒度，前缀复用率最大化
- 天然支持分叉（同一 system prompt 下的不同用户）
- 对多租户 Agent 场景（N 个用户 × 1 个 system prompt）收益极大

缺点：
- 树结构的维护成本（插入、驱逐、分裂节点）
- 内存管理更复杂（树节点生命周期与 block 不对齐）
- 实现复杂度高 2-3 倍
```

#### 选型判断（基于实测）

| 场景 | 推荐引擎 | 理由 |
|------|---------|------|
| 单租户、短对话 | vLLM | 简单够用，block hash 缓存命中率已够高 |
| 多租户（N用户共 system prompt） | SGLang | RadixAttention 对共享前缀 batch 天然优化 |
| RAG（检索结果每次不同） | vLLM | 前缀缓存意义不大，不如用 continuous batching |
| 长上下文多轮对话 | SGLang | 历史轮次 = 天然前缀，复用率极高 |

### 2.4 MLA 再深入：为什么是架构级优化

标准 GQA（Grouped Query Attention）减少 KV heads 数量来降低存储：
```
MHA: K/V 各 H 个 head → 存 2 × H × d_head per token
GQA: K/V 各 G 个 group（G < H）→ 存 2 × G × d_head per token
方法：减少 K/V 的独立副本数
```

MLA（Multi-head Latent Attention，DeepSeek V2/V3）则是压缩维度：
```
将 K, V 投影到低维 latent 向量 c，只存 c 而不存完整 K/V：
c = W_c × x    （低维，比如 512 维）
K = W_k × c    （在 attention 时实时重建）
V = W_v × c    （在 attention 时实时重建）

存储：只需存 c（512维 × FP16 = 1 KB/token）
vs 标准：K+V（2 × 128头 × 128维 = 32 KB/token）
→ 内存节省 32x（理论），实测 5-13x
```

**关键洞察**：MLA 的代价是 attention 时需要两次矩阵乘（先重建 K/V，再计算 attention），但这是计算换内存，在 memory-bandwidth-bound 的 decode 阶段是合算的。

## 三、结合现有项目的反思

### 反思 A：我们的 Prompt 到底有多少能被缓存？

前几天（05-04）已经提出要做 prompt caching 审计。今天读了 vLLM 的 block-level prefix caching 文档，更深刻理解了一个细节：

**问题：** vLLM 的 cache 单位是 16-token block。如果我们的 system prompt 是 1000 token，但每次调用时前面还有动态的 `用户上下文`（比如用户姓名、当前时间），那么 block hash 会立刻失效——整个前缀都无法复用。

**行动：**
- 检查调用链：`system prompt` 里有没有混入动态变量？
- 如果有：把静态部分（角色定义、规则）提到最前面，动态部分放到 user message 里
- 这不只影响 API 侧的 prompt caching，更影响自托管时 vLLM/SGLang 的前缀复用率

### 反思 B：多租户 Agent 场景应考虑 SGLang

目前架构是 API 模式（Anthropic/OpenAI），所以推理引擎无法选择。但假设未来自托管：

- 如果是**单个 AI 助手**服务多个用户（相同 system prompt），SGLang RadixAttention 的收益是结构性的
- 估算：100 用户 × 2000 token system prompt = 如果能复用，每次请求省 2000 token prefill
- 按 Llama-70B prefill throughput ~500 tok/s，节省 4 秒/请求 TTFT

### 反思 C：H2O 的场景适配

我们的 Agent 场景是否适合引入 H2O（Cache Eviction）？

| 我们的场景 | H2O 适用度 | 原因 |
|----------|----------|------|
| 短对话 (<2K token) | ❌ 不需要 | Cache 不会满，驱逐无意义 |
| 长文档摘要 RAG (>8K) | ✅ 适用 | 内容可 evict，关键词是 heavy hitter |
| 代码生成 Agent | ⚠️ 谨慎 | 函数签名/变量定义需要精确引用，evict 可能出 bug |
| 多轮规划 Agent | ❌ 不推荐 | 每一轮的中间决策都可能是关键依赖 |

**结论：** H2O 只在长文档生成类场景有净收益。代码和推理场景宁可用 TurboQuant 压缩 + MLA 架构，而不是 eviction。

### 反思 D：KV Cache 优化的决策树

基于今天的学习，整理一个选型决策树：

```
场景分析 →

Q1: 上下文有没有共享前缀（同一 system prompt 多用户）？
  是 → Prefix Caching（API 侧用 cache_control，自托管用 SGLang）
  否 → 进 Q2

Q2: 单次 context 长度有多长？
  < 8K → 标准 KV，主要靠 batching 优化
  8K-64K → 考虑 TurboQuant（3.5-bit KV 压缩）
  > 64K → Cache Eviction（H2O，选择适用场景）+ 压缩

Q3: 对精度敏感度高吗？
  高（代码生成/精确问答）→ 压缩 > 驱逐，MLA 架构
  低（摘要/翻译）→ 激进 eviction 可接受

Q4: 是否自托管？
  是 → 推理引擎选型（场景决定 SGLang vs vLLM），开启 FP8/Marlin kernel
  否 → 只能优化 prompt 结构 + cache_control 标记
```
