## 🧠 核心框架：三类 Agent

| 类型 | 特征 | 典型工具 | 建设周期 |
|------|------|----------|---------|
| **Category 1：确定性自动化** | 流程固定，AI 处理局部内容 | n8n, Zapier, Make.com | 4-6周 |
| **Category 2：推理行动型 Agent** | AI 自主决定下一步，工具调用 | LangGraph, CrewAI, Cursor | 2-4月 |
| **Category 3：多 Agent 网络** | 多个专属 Agent 协同 | ADK, AutoGen | 4-6月+ |

**最大的坑：** 用 Category 2 的框架去解决 Category 1 的问题。过度工程，贵且慢。反过来用 Category 1 解决 Category 2 的问题，生产环境直接崩。

## 💡 我的思考与观点

### 1. "Agent 爆炸"时代，PM 的新核心能力是**分类判断**

以前 PM 最重要的能力是需求拆解和优先级排序。现在这还不够——你必须能先判断"这个 AI 想法架构上是什么类型"。

这篇文章给了一个非常实用的 5 秒自检：**"能不能用流程图穷举所有路径？"**
- 能 → Category 1，别废话上 n8n
- 不能 → Category 2，准备 Agent 框架
- 需要多个专业 Agent 协同 → Category 3，先问自己凭什么

我觉得这个分类背后还有一个更本质的问题：**可预测性**。Category 1 是"用户意图已知、路径已知"；Category 2 是"意图已知、路径未知"；Category 3 是"意图本身也可能是模糊的"。

### 2. 大多数"AI 功能"提案其实是 Category 1，但团队喜欢把它包装成 Category 2

为什么？因为 Category 2 听起来更酷。"我们要做一个自主 Agent" vs "我们要做一个自动化流程"——前者更容易拿到预算、发 PR 稿。

这其实是一种**技术债的反向伪装**：把简单问题搞复杂，然后让复杂度显得正当。

作为 PM，要有足够的勇气说"我们做 n8n 流程就够了"，而不是追逐看起来更性感的架构。

### 3. 三类 Agent 的 ROI 曲线完全不同，但团队经常混用同一个成功指标

这是最危险的地方。Category 1 的 ROI 应该在 6-8 周内可见；Category 2 可能要 3 个月才能看到稳定表现；Category 3 的价值要跨季度来算。

如果用 Category 1 的验证节奏去评估 Category 2 的项目，很容易"觉得没效果就砍掉"——其实可能只是还没爬过学习曲线。

## 🔗 结合我们的产品反思

### 旅行 App
今天学到的框架让我重新审视"AI 行程规划"这个功能：

- **"帮我规划5天东京行"** → Category 1 完全够，流程可画，路径有限
- **"帮我找一个这个季节适合内向者的冷门目的地"** → Category 2，意图清晰但路径开放
- **"帮我同时协调多人行程偏好 + 价格 + 时间"** → 接近 Category 3，但我们现阶段不应该碰

**重要结论：我们把旅行规划定义为 Category 1，但 UI 要包装成像 Category 2——体验上有智能感，成本和周期按 Category 1 控制。这才是 PM 的真正价值点。**

### 英语学习 App
- **个性化内容推荐** → Category 1（触发逻辑清晰：学了什么词 → 推相关内容）
- **开放式口语练习对话** → Category 2（用户输入完全不可预测）
- **智能语法纠错反馈** → Category 1（规则+模型，可以流程图画出来）

**问题：** 如果我们把"对话练习"作为核心功能投入，就必须接受 Category 2 的成本和周期，不能用 Category 1 的节奏催进度。这个认知对齐要在研发前做，不能事后解释。

### 云间 App
- 日记写完后的「今日总结」→ Category 1，触发条件明确
- 「帮我找到过去一年的情绪低谷规律」→ Category 2，路径开放但意图清晰
- 联合日记 + 日历 + 习惯数据做综合洞察 → 目前对我们来说是 Category 3，先不碰
